## 1. The thesis

*Boost Your Django DX* by Adam Johnson is a guided tour of tools and habits that improve **DX (Development Experience)** when working on a Django project. The book is framed by **kaizen**, the Japanese philosophy of constant improvement to a process — treating every bug or friction as feedback about where the system can be revised. The focus is on tools and practices rather than writing "normal" Django code, and on open source, cross-platform tools that apply broadly to Django projects rather than third-party packages or production environments.

The book is not a Django tutorial. It assumes the reader already builds Django sites and instead recommends a particular setup: which packages to install, which configuration files to commit, which Git hooks to enforce, which patterns to avoid. Several of the recommended packages were written by Johnson himself, who is candid about which ones he reaches for and why. The chapters are roughly ordered to follow the life of a project but stand alone and can be read in any order. Examples target Python 3.12 and Django 5.0.

## 2. Finding information faster

Code begins with knowledge, and the speed of looking things up is a real ceiling on coding speed. The first practical chapter focuses on documentation tools.

The single biggest recommendation is **DevDocs**, a free browser-based aggregator that ships fast fuzzy search across many documentation sources (Python, Django, MDN, Git, the major databases, Jinja, Django REST Framework, and more) and works offline. After a one-time enable of the relevant docsets — DevDocs stores configuration in browser local storage — `/` focuses the search box; typing a partial term such as `integf` fuzzy-matches `IntegerField`, and arrow keys move through results. Limiting search to a single docset is two keystrokes (`dj` + `tab`); `escape` resets. An "Install all" button under the kebab menu downloads every enabled docset for true offline use. DevDocs is browser-based and universal; **Dash** (paid, macOS only) and its open-source clone **Zeal** are the original desktop equivalents.

For general web searches, the book recommends **DuckDuckGo** as a developer-friendly default — its results are roughly 95% as good as Google's for general queries and "nearly always good enough for development", and the gap is recoverable via the `!g` bang. DuckDuckGo's three developer-focused features are full keyboard navigation (`/` or `h` focuses the search box; `j`/`k` move between results; `ctrl+enter` opens in a background tab — Google notably has none of these), **bangs** (exclamation-mark prefixes that redirect a query directly to another search engine without an intermediate page — `!py sorted` jumps straight to the Python documentation search for `sorted`), and **Instant Answers** that surface Python documentation extracts or top StackOverflow answers beside results. The most useful bangs for Django work are `!py`/`!python`, `!dj`/`!django`, `!drf`, `!pep`, `!pip`, `!djpackages`, `!djticket`, `!gh`, `!ghc`, and the various database bangs.

A handful of community-built Django reference sites complement the official docs: **Classy Class-Based Views** (ccbv.co.uk) and its siblings **Classy Django Forms** and **Classy Django REST Framework** are class explorers for the respective hierarchies, **Template Tags and Filters** (djangotemplatetagsandfilters.com) is a cheat sheet for the template language, and **Awesome Django** (awesomedjango.org) is a vetted index of further resources.

For documentation that DevDocs doesn't carry and that has no official download, **Wget** can mirror a whole site for offline browsing in a single command:

```console
$ wget --mirror --convert-links --adjust-extension --page-requisites --no-parent <website>
```

The flags activate full-site mirroring, rewrite links relative, add file extensions, pull in CSS/images, and prevent recursion to parent paths. Some downloaded sites use AJAX or JS navigation that browsers block under `file://`; serving them through Python's stdlib HTTP server (`python -m http.server 8001` — port 8001 avoids colliding with `runserver`'s default 8000) fixes the issue.

The chapter closes with per-source navigation tips. Python documentation has stable, minor-version, and patch-version URL forms (`/3/`, `/3.10/`, `/release/3.10.0/`); always link to `/3/` in code comments so links survive Python upgrades. The FAQ, Glossary, and HOWTOs are underused alongside the reference. Django documentation has a hidden `stable` URL specifier — `https://docs.djangoproject.com/en/stable/...` — that doesn't appear in the version switcher but is the right form for evergreen links. Read the Docs offers cross-version search narrowable by *Code API Type* and a `<project>.rtfd.io` shortcut domain. **Sphinx**-built docs (the de facto standard, including Python's and Django's) embed import paths in heading anchors — clicking the pilcrow next to a definition reveals `html.parser.HTMLParser` in the URL fragment, which is also the import path.

## 3. The foundation: virtual environments and dependencies

Johnson recommends sticking to the default Python tools — venv plus Pip — over Conda, Poetry, or Pipenv. Every Python developer has some familiarity with them, Pip is the universal installer with the broadest edge-case coverage, and fewer layers means fewer things to debug when an environment breaks. One small habit matters: always invoke Pip as `python -m pip`, never bare `pip`, because the bare command can resolve to a different Python than the active one and silently install into the wrong environment. Aliasing `pip='python -m pip'` in `~/.bashrc` or `~/.zshrc` makes the safe form the default.

The recommended `venv` create command is `python -m venv --prompt . --upgrade-deps .venv`: `.venv` is the conventional hidden directory name (auto-detected by VS Code and PyCharm), `--prompt .` labels the shell prompt with the project directory, and `--upgrade-deps` upgrades Pip inside the env so the "new release available" nag stops. Virtual environments must never be committed; Python 3.13 onward writes a `.gitignore` inside new envs automatically, but earlier versions need a manual `/.venv/` line. If a venv has been committed, `git rm -r --cached .venv` stops tracking without deleting from disk.

The original third-party **virtualenv** package is worth knowing: it was copied into the standard library as venv in Python 3.3 (PEP 405) and has continued to evolve while venv has mostly stood still. It's roughly ten times faster (~0.3s vs ~3s) and seeds new envs with a cached recent Pip — worth installing for those who create environments often.

Dependency pinning is the larger topic of the chapter. The naive approach — `pip freeze > requirements.txt` — has three problems: it mixes top-level and transitive dependencies with no way to tell them apart, new direct dependencies don't auto-pin their transitives, and editing a version doesn't recompute transitives. The recommended replacement is **pip-tools**' `pip-compile`, which reads a hand-maintained `requirements.in` (the project's direct dependencies, mostly unconstrained) and produces a fully pinned `requirements.txt` with `# via` annotations explaining each entry. Both files are committed; this is the same pattern npm and Cargo follow. To add a dependency: append to `.in`, run `pip-compile`, install. To upgrade a single package: `pip-compile -P <pkg>`. To upgrade everything: `pip-compile -U`. The `pip-sync` command brings the env into exact alignment with the lockfile, useful for cleaning out stale packages.

Even with pinning, environments drift when teammates pull new requirements but forget to reinstall. The `pip-lock` package (one of Johnson's own) wires a `check_requirements()` call into `manage.py`, so every Django command compares the installed packages against `requirements.txt` and exits with a friendly "run `python -m pip install -r requirements.txt`" message if they don't match.

The chapter also offers a heuristic for evaluating new dependencies: the **Well Maintained Test**, a list of twelve yes/no questions (production-ready claim, sufficient docs, changelog, responsive bug reports, sufficient tests, tests on the latest language version, tests on the latest integration version, CI configuration, CI passing, well used, recent commit, recent release). It's a guide for judgement, not a hard pass/fail; each "no" is a chance to contribute. On upgrade cadence, Johnson likens dependency upgrades to washing dishes: defer too long and they consume all available time. Every two weeks to once a month works for medium-sized teams; the calendar anchors are Python's October release, Django's eight-month feature release cycle (every third is LTS, version ending `.2`), and PostgreSQL's annual October release. He recommends upgrading every Django feature release rather than only LTS — non-LTS feature versions are just as stable but have shorter support windows, and upgrading regularly keeps the practice routine.

The final tool here is **Python's development mode** (`python -X dev` or `PYTHONDEVMODE=1`), an opt-in mode that surfaces `DeprecationWarning`, `PendingDeprecationWarning`, `ImportWarning`, `ResourceWarning`, the `faulthandler` module, C-level debug hooks, and `asyncio` debug mode. The `asyncio` mode is the headline for Channels users — it flags missing `await` calls. Verify it's active with `python -c 'import sys; print(sys.flags.dev_mode)'`. The book later shows a Django system check that enforces it.

## 4. Tightening the interactive loop

### 4.1. The shell

`./manage.py shell` launches a Python REPL pre-configured with the project, but the stdlib REPL is bare. Installing **IPython** (Django's `shell` auto-detects it) replaces `>>>` with `In [1]:` and adds three categories of feature.

The first is help and autocomplete: `name?` shows type, signature, and docstring; `name??` adds the source code; tab completion handles dictionary-key expansion (`d["us<tab>` becomes `d["user_count`). The second is history: `Out[n]` and the shortcut `_n` retrieve any previous result, `In[n]` retrieves the source string, and `%hist` prints prior inputs without prompts — ideal for copying a session into a file. `%rerun 2 1 3` re-executes specific lines in a chosen order. The third is paste handling: IPython treats a paste as one block (so blank lines inside a function don't terminate it), and `%cpaste` opens a paste prompt that ends on `--`, useful when SSH'd into a remote shell.

`%load_ext autoreload` plus `%autoreload 2` re-imports modified modules before every prompt, so edits take effect without restarting the shell. Caveats: registry patterns (Django admin, `atexit` handlers) re-fire on each reload, and reloading models triggers `RuntimeWarning: Model 'X' was already registered`. Single-module work is fine; iterating across many model files is risky.

`IPython.embed()` drops an IPython prompt at any point in the code — inside a view, inside a management command — with full access to local variables. When embedding inside a view, run `runserver --nothreading --noreload` to avoid concurrent re-entry and file-change restarts. For debugging, `%debug` opens IPython's pdb-extended debugger on the line of the last exception. To get the same upgrade outside IPython, install **ipdb** and set `PYTHONBREAKPOINT=ipdb.set_trace` so plain `breakpoint()` calls open the IPython debugger; Django's `--pdb` test flag picks `ipdb` up automatically. The pytest equivalent is `addopts = --pdbcls=IPython.terminal.debugger:TerminalPdb`.

For micro-benchmarking, `%timeit <expr>` auto-determines a sensible loop count and reports mean ± std. dev. with units. A throwaway example used in the book — `%timeit dict()` (~36.5 ns) versus `%timeit {}` (~16.7 ns) — illustrates that the literal form skips the name lookup `dict()` requires, since `dict` could be locally rebound.

Finally, **django-read-only** addresses the production-shell hazard: running `manage.py shell` against production accelerates feature work but it's easy to mix up terminal sessions and run a destructive command meant for dev. The package (added to `INSTALLED_APPS`) makes the database read-only when `DJANGO_READ_ONLY=1` is set, raising `DjangoReadOnlyError` on any write. `django_read_only.enable_writes()` and a `temp_writes()` context manager are escape hatches. The recommended setup exports the env var in the production shell's `.bashrc` so protection is on by default.

### 4.2. The development server

**django-debug-toolbar** is rated a "must-have". The package injects a floating "DjDT" tab into every HTML response — only when `DEBUG=1` — and clicking it opens a panel that exposes per-request data Django would otherwise hide. The Versions panel reports the Python/Django/package versions; the Time panel splits in-Django CPU time (User CPU ≈ Python time, System CPU ≈ DB I/O) from browser-side timings; the Templates panel lists every rendered template (including those pulled in via `{% extends %}` and `{% include %}`) and reveals the chained context dictionaries; and the History panel allows the toolbar to attach to non-HTML responses (open the JSON endpoint, then go back to a toolbar-bearing page, click "Refresh" then "Switch" on the API request).

The SQL panel is the most consequential. It lists every query in order with timeline blocks colour-coded slowest-red to fastest-green. `+` expands to the full SQL, `Sel` re-runs SELECTs, and `Expl` runs the database's `EXPLAIN`. The book walks through a concrete N+1 example: a `Books` page with 96 queries (95 similar, 95 duplicates), where the outer `{% for book in books %}` runs one query for books and the inner `{{ book.author.name }}` triggers one query per book. Switching `Book.objects.all()` to `Book.objects.prefetch_related("author")` collapses the page to two queries (one for books, one batched `WHERE "core_author"."id" IN (...)`) and ten times the speed.

**django-browser-reload** (another Johnson package) eliminates the manual edit-save-refresh loop. It uses Django's existing change detection plus a small JS shim: template changes fire reloads within milliseconds; static asset changes reload the page; Python code changes wait for `runserver` to finish restarting before reloading, so the browser doesn't refresh into an "unable to connect" error. It only operates when `DEBUG = True`, so it's safe to keep installed in production.

**Rich** improves terminal output for both server logs and management commands. Wiring `rich.logging.RichHandler` into Django's `LOGGING` (with `rich_tracebacks=True`, `tracebacks_show_locals=True`, and the `require_debug_true` filter) gives colour-coded log levels, parsed request log lines, clickable file:line links on capable terminals, and rich tracebacks (clickable file references, syntax-highlighted source extract, table of local variables) — Django's debug error page, but in the terminal, and working for exceptions outside HTTP requests. The companion **django-rich** package provides a `RichCommand` base class for management commands; subclassing it gives `self.console` for printing tables, progress bars, and other Rich primitives.

## 5. Automated code quality

### 5.1. The pre-commit framework

Before tools, the chapter recommends **EditorConfig**: a cross-editor standard (`.editorconfig` at the repo root) that propagates encoding, line endings, indentation, trailing-whitespace rules, and final-newline rules to every editor on the team. The recommended baseline is UTF-8 with LF line endings (LF even on Windows), 2-space indentation by default, 4-space override for `.py`, trailing whitespace trimmed, and a final newline inserted. Trailing whitespace creates "diff noise"; missing final newlines violate POSIX. EditorConfig only nudges editors going forward — pre-commit hooks `trailing-whitespace` and `end-of-file-fixer` enforce the rules retroactively.

**pre-commit** is the framework that anchors everything else. It installs as Git's `pre-commit` hook and runs configured tools against staged files before each commit. Hooks live outside `.git/` (so they're shared via the repo), tool installation and version pinning are automated, and multiple tools can run in order. Created by Anthony Sottile in 2014; used by Django since 2020 and CPython since 2023. Install once globally (`brew install pre-commit` or `pip install` to system Python), run `pre-commit install` per repo, and `pre-commit sample-config` produces a starter config. On adoption, run `pre-commit run --all-files` in one big commit to fix existing files before turning the hook on.

The configuration file lists `repos` of hooks pinned by Git tag. `default_language_version` pins Python and Node versions to avoid behaviour drift across the team. The everyday commands are `git commit` (runs hooks against staged files), `git commit -n` (skip — for committing broken code on a feature branch), `SKIP=hook1,hook2 git commit` (skip selected hooks), `pre-commit run` (against staged), `pre-commit run --all-files` (the CI command, since devs can `-n` past hooks locally). When a hook fix conflicts with a stashed change, pre-commit discards the fix — hand-authored work outranks automation. `pre-commit autoupdate` bumps every hook's `rev` to the latest tag (but does not touch `additional_dependencies`, which need manual bumps). The hosted **pre-commit ci** service auto-fixes on PRs, runs `autoupdate` weekly, and reuses the hook-environment cache for speed; a free "lite" version runs in GitHub Actions but is slower.

### 5.2. Formatters and linters

**Black** is the dominant Python formatter — a single style based on PEP 8 with no configuration knobs (the name nods to Henry Ford's Model T quip, "any color so long as it is black"). 88-character lines, 4-space indents, double quotes, call chains split with one method per line and dots at line start (great for the Django ORM), `\` line continuations replaced with parens. The only configurable lever is the **magic trailing comma**: a comma after the last collection item signals "expect this to grow" and forces one-item-per-line layout, keeping future diffs minimal. Pin `target-version` in `pyproject.toml` to prevent per-file syntax detection from triggering reformatting. Adopt in one big commit and configure Git's "ignore commits in blame" mechanism so the formatting churn doesn't poison `git blame`. **blacken-docs** applies Black to code snippets inside reStructuredText, Markdown, LaTeX, and `.py` docstrings — a side benefit is that it catches broken example code via Black's syntax errors.

**isort** sorts and groups imports (future, standard library, third party, first party, local relative — `import` before `from`, alphabetized). Use the `black` profile in `pyproject.toml` so isort's output is Black-compatible; the legacy `django` profile is historical (Django itself moved to the `black` profile in 2022). The `add_imports` / `remove_imports` features inject or strip an import in every file — useful for `from __future__ import annotations` rollouts and rollbacks. **reorder-python-imports** is an alternative sorter, also by Anthony Sottile, with one key difference: it forces one `from` import per line, dramatically reducing merge conflicts because concurrent edits no longer touch the same line. (isort can mimic this with `force_single_line = true`.)

**Flake8** is the linting framework. By default it pulls from pyflakes, pycodestyle, and mccabe; a rich plugin ecosystem stacks on top. Linters catch what formatters can't — undefined names, unused imports, possible bugs that need human judgement. Set `max-line-length = 88` and `extend-ignore = E203` to align with Black. Place Flake8 after formatters in the pre-commit chain. Plugins install by adding the package to `additional_dependencies`. The most useful are **flake8-bugbear** (extra bug checks), **flake8-no-pep420** (bans implicit namespace packages, which silently disable test discovery and coverage), **flake8-comprehensions**, **flake8-tidy-imports**, **flake8-print**, and **flake8-logging**.

**Mypy** is a static type checker that catches mismatched types via Python's annotation syntax. Its understanding of code structure goes deeper than linters can. **django-stubs** adds Django-aware hints. Mypy in pre-commit runs in pre-commit's separate virtualenv (use `mirrors-mypy`); if that breaks, fall back to a `system` hook that runs Mypy in the project's venv. The book defers depth — type-checking is its own topic — and points readers at Mypy's getting-started guide and the awesome-python-typing list.

### 5.3. Modernizers and adjacent languages

**pyupgrade** (Anthony Sottile, 2017) rewrites old Python syntax into the current preferred form; pass `--py312-plus` (or the appropriate flag for the target Python version). Examples: `'{foo}'.format(foo=foo)` becomes an f-string; `super(C, self).f()` becomes `super().f()`; `if x is 5` becomes `if x == 5`. **django-upgrade** (Adam Johnson, 2021) is the same idea for Django: `args: [--target-version, "4.0"]` rewrites code to avoid deprecation warnings, e.g., `django.conf.urls.url(r'^$', ...)` → `django.urls.path('', ...)`. Order modernizers before formatters and linters — they emit code that may need cleanup by Black and shouldn't trip Flake8 along the way.

**DjHTML** (Jaap Joris, 2021) is a Django template formatter that *only* corrects indentation, deliberately conservative to avoid breaking templates. Default is 4-space indent; override to `--tabwidth=2` to match the de facto HTML/CSS/JS standard. Verify with `git diff --ignore-all-space` after — `<pre>` and elements with `white-space: pre` in CSS are sensitive to whitespace shifts.

For the JS/CSS side: **Prettier** is "Black for JavaScript", a multi-language formatter covering JS/JSX/TS, CSS/Less/SCSS, HTML, JSON, GraphQL, Markdown, and YAML. It cannot format Django HTML templates because the template syntax isn't valid HTML; scope it via `types_or` to safe types. Use `mirrors-prettier`; for plugins, set `additional_dependencies` *and re-declare prettier itself there* because the mirror repo replaces its own declaration. **ESLint** is the dominant JavaScript linter; the recommended ruleset (`@eslint/js`'s `js.configs.recommended`) is the starting point because without it ESLint runs no rules. Use the new `eslint.config.js` "flat config" format from ESLint 9 in a `repo: local` pre-commit hook (the older `mirrors-eslint` doesn't support flat config). Place ESLint after Prettier so it doesn't flag formatting Prettier would have fixed; **eslint-config-prettier** disables the ESLint rules that conflict with Prettier's choices. **ShellCheck** lints shell scripts — POSIX sh, Bash, dash, ksh — catching the many sharp edges in shell syntax. Use `shellcheck-py` to install it via pip and sidestep cross-platform Haskell-toolchain issues.

The default `pre-commit-hooks` repository ships further cheap checks worth adding: `check-case-conflict` (files differing only by case, broken on macOS HFS and Windows FAT), `check-json` / `check-toml` / `check-xml` / `check-yaml`, `check-merge-conflict` (leftover Git markers), `check-added-large-files`.

## 6. Building your own tools

Pre-commit ships two "virtual languages" for writing small custom hooks inline in `.pre-commit-config.yaml` without packaging a separate tool. Both go under `repo: local`. **`language: fail`** automatically fails on every matching file with `entry` printed as the message — useful for banning specific files (after migrating fixtures from JSON to YAML, a hook on `files: /fixtures/` with `exclude_types: [yaml]` prevents JSON fixtures from creeping back). **`language: pygrep`** fails files whose content matches a Python regex — useful for forbidding patterns inside files (a hook on `types: [html]` with `entry: novalidate` catches `<form novalidate>` left in after testing server-side validation).

When pygrep is too crude and an existing Flake8 plugin doesn't exist, write a local Flake8 plugin. Local plugins skip the Python packaging dance — start local, package later if useful. Flake8 plugins receive Python's **AST** (Abstract Syntax Tree), the parser's output stripped of formatting and comments. Explore with `ast.parse('1 + 1')`, which yields a `Module` whose body is a `BinOp(left=Constant(1), op=Add(), right=Constant(1))`. `ast.walk(node)` is the workhorse: a generator over every node in a tree, paired with `isinstance(node, ast.X)` checks for finding patterns. A plugin is a class with `name`, `version`, `__init__(self, tree)`, and a `run` generator yielding `(lineno, col_offset, "CODE message", type(self))` per error. The book's example bans `lambda` (Guido van Rossum once proposed removing it from Python 3) by walking the tree and yielding on every `ast.Lambda`. Register in `setup.cfg` under `[flake8:local-plugins]` with `extension = L = example.ext.flake8:LambdaChecker` and `paths = .`.

When even Flake8 isn't the right host, write a small standalone Python script. Place it in `scripts/` or `bin/` (not inside the project package, since it's not part of the importable API). Pre-commit expects four things: tools accept filenames as positional arguments, return non-zero on any change (so output is shown even after auto-fixes), print to stdout/stderr, and modify files in place (pre-commit detects changes via Git). The book's worked example is a fixer that rewrites versioned Django doc URLs (`docs.djangoproject.com/en/5.0/...`) to the evergreen `/stable/` form, using `argparse.FileType("r+", encoding="utf-8")` to open files for read/write, `re.subn` to count substitutions, and `seek/write/truncate` to rewrite in place. Wire it into pre-commit with `language: script`, `entry: scripts/fix_django_links.py`, and `types_or: [markdown, python, rst]`. Test with pytest's `tmp_path` and `capsys` fixtures: write an example file under `tmp_path`, call `main([str(example)])`, assert exit code is 1 and the file content was rewritten.

## 7. Settings

Johnson recommends a single settings file driven by environment variables over the common multi-file split (`settings/dev.py`, `settings/prod.py`). One file is more straightforward, keeps configuration separate from code, and lets a deploy be reconfigured without a code change. Environment variables are universal across OSes and platforms — popularized by Heroku's "Twelve-Factor App" — and can be loaded locally from a `.env` file via **python-dotenv**'s `load_dotenv()` call at the top of settings. The `.env` file holds secrets and must never be committed; ship a tracked `.env-example` template with placeholder values and comments instead.

The recommended settings layout is a five-section header structure: Setup (imports, `.env` loading), Django Core Settings (from Django's "Core Settings" reference), Django Contrib Settings (one sub-section per `django.contrib.*` app, alphabetized), Third Party Settings (one sub-section per package, alphabetized), and Project Settings (custom settings, grouped per app if multiple). Alphabetize within each group; hoist dependencies like `BASE_DIR` to the top of their section. Order `INSTALLED_APPS` First Party → Third Party → Contrib, because Django resolves ties by first-listed-wins, so apps that may need to override resources from others should come first. `BASE_DIR` itself is a pseudo-setting (Django doesn't read it, but the default `settings.py` defines it and many third-party packages reference it); since Django 3.1 the modern form uses `pathlib`: `BASE_DIR = Path(__file__).resolve().parent.parent`.

Tests pose two problems: a stray `.env` file could leak local config (worst case, hit a real remote API) and some settings need to differ during tests for speed or isolation. Skip `.env` loading by guarding `load_dotenv()` against the test command. Override settings via a custom `DiscoverRunner` subclass that wraps `run_tests()` in `override_settings(**TEST_SETTINGS)` (Django) or via an `autouse=True`, `scope="session"` fixture in top-level `conftest.py` (pytest).

The chapter spends substantial space on settings anti-patterns. Don't read settings at import time — `page_size = settings.PAGE_SIZE` at module level captures the value once, so `override_settings` won't take effect; read inside the function or method. Don't mutate `settings.X = ...` directly — it bypasses the `setting_changed` signal so receivers (cache rebuilds, etc.) aren't notified; always use `override_settings`. Don't import the project's settings module — `from example import settings` bypasses Django's settings machinery; always `from django.conf import settings`, enforced via `flake8-tidy-imports` `banned-modules`. Don't promote constants to settings (a value that can't change isn't a setting). Don't define dynamic constants (env-var reads in random modules) outside settings, because it makes the project's environment variables undiscoverable, makes timing unpredictable, and `override_settings` can't reach them — centralize all env-var reads in the settings file. Name custom settings verbosely (`EXAMPLE_API_TIMEOUT_SECONDS`, not `EA_TO`) since settings share a flat namespace. Override complex (dict-shaped) settings with `settings.EXAMPLE | {"PAGE_SIZE": 2}` rather than `override_settings(EXAMPLE={"PAGE_SIZE": 2})`, which erases sibling keys.

It's fine to put logic and helper functions in the settings file when those functions are only used there. Test such logic: import the settings module directly (one of the few legitimate exceptions to the no-import rule); use `RequestFactory` for callable settings like django-debug-toolbar's `SHOW_TOOLBAR_CALLBACK`. For module-level branches (`if os.environ.get("HTTPS") == "1": ...`), exercise them by re-importing the module — copy `import_fresh_module()` from Python's `test.support` (it has no public-API guarantees, so vendoring is safer than depending on it) and combine with `mock.patch.dict(os.environ, {...})`.

## 8. Models, data, and migrations

Local databases need realistic data. Production data — even pseudonymized — risks re-identification through correlations and may be illegal under GDPR; the book recommends generated data via Django ORM code, which is flexible, adaptable, and easy to test (more so than SQL backups or fixture files). Put the seeding code in a management command named `seed_database` so a fresh local DB is two steps: `migrate` then `seed_database`. Wrap `handle()` in `@transaction.atomic` so a partial run leaves no rows behind. Guard against accidental production runs by checking `if Author.objects.exists()` early and raising `CommandError` (Django prints it in red and exits non-zero — the idiomatic way to fail a management command). Use `bulk_create()` for related batches. The sites framework's default `Site` is created by its migration, so `seed_database` can update it (e.g., to `localhost:8000`) inline rather than via a separate data migration.

Test the command by wrapping `call_command()` in a small helper that captures `stdout`/`stderr` via `StringIO`. To catch the case where a new model is added and someone forgets to seed it, loop over `apps.get_app_configs()` (filtered by name prefix to skip third-party/contrib apps), iterate each `AppConfig.get_models()`, and assert `model.objects.exists()`. Wrap each assertion in `subTest()` so one missing model doesn't mask others. Maintain an `exempt_models` set of `(app_name, model_name)` tuples for models intentionally not seeded, with the failure message naming the offending model and pointing at this set.

Hand-filling each model's fields scales poorly. **Factory Boy** automates it: a factory class fills fields with sensible defaults so callers specify only the values that matter for the current call. Subclass `DjangoModelFactory`, set the target model in inner `class Meta`, declare each field, and place factories next to `models.py` as `<app>/factories.py`. **Faker** integration generates plausible random strings: `Faker("name")`, `Faker("sentence")`. `SubFactory` chains foreign keys — calling `BookFactory()` invokes `AuthorFactory` automatically, so the `Book` and its `Author` come together; override sub-fields with Django-style double-underscore syntax: `BookFactory(author__name="Bruce Wayne")`. `BookFactory()` returns a saved model instance (not a factory instance); `BookFactory.create_batch(5)` makes five at once; `BookFactory.build()` returns an unsaved instance with `id is None`, useful for tests that don't touch the database. In tests, factories collapse setup; in `seed_database`, they shrink the command. **Model Bakery** is a Django-specific alternative; the author still recommends Factory Boy for its broader feature set.

Two migration safeguards round out the chapter. First, run `makemigrations --dry-run --check` as a unit test, not just in CI. Django requires migrations for every model field and `Meta` change, even non-DB ones like `Field.choices`; forgetting one ships code that crashes on deploy. The test calls `call_command("makemigrations", "--dry-run", "--check", ...)`, catches the `SystemExit` raised on failure, and re-raises it as `AssertionError("Pending migrations:\n" + out.getvalue())` — copy-paste-ready into any project.

Second, **django-linear-migrations** prevents merge migrations entirely. Django's built-in fix for branched migration histories — `makemigrations --merge` creating a `0003_merge` that depends on both leaves — has three problems: each merge migration is a fix after the fact (the conflict surfaces only after merging to main and breaking CI), it doesn't enforce the same execution order across environments (staging may run migrations in commit order while production runs them in the merge migration's declared order, dangerous with `RunSQL`/`RunPython`), and it complicates rollbacks. The package, written by Johnson after seeing many in-house versions of the same idea, has every app keep a `max_migration.txt` file with the name of its latest migration. When two branches each touch it, Git produces a merge conflict, forcing migrations to be reordered linearly *before* merging. It's compatible with apps that already contain historical merge migrations — it enforces linearity from install onwards. The bundled `rebase_migration` command automates the simple case of editing the new migration's `dependencies` and `max_migration.txt` to relinearize.

## 9. The system checks framework

Django's **system checks** framework is a mechanism for collecting validation functions that each return a list of `CheckMessage`s; the framework runs them all, displays the messages, and aborts the current command if any are errors. Built-in checks cover incompatibilities, security, and uninstalled app dependencies; third-party packages plug in to validate their own configuration. Checks fire at the start of most management commands, on every `runserver` reload, and via `./manage.py check`. The `--deploy` flag enables an extended set of checks targeted at production environments — run it before `migrate` in deploys to exit early on misconfiguration. To suppress a check, list its ID (e.g., `"models.E034"` for the 30-character Oracle index name limit) in `SILENCED_SYSTEM_CHECKS`; always add an explanatory comment, since silencing removes a safety feature. The advantages of checks over ad-hoc validation are batch reporting (one pass surfaces every issue), early firing in dev workflows, and the ability to inspect live runtime objects in ways linters can't. The trade-off: checks must be fast — milliseconds, not seconds — so no file reads, DB queries, or source parsing.

Custom checks are functions with the signature `def my_check(*, app_configs, databases, **kwargs)` returning a list of `CheckMessage`s. Always accept `**kwargs` for forward compatibility. The five severity subclasses are `Debug`, `Info`, `Warning`, `Error`, and `Critical`; in practice almost all custom checks use `Error` (fails the run) or `Warning` (displayed but doesn't fail). A `CheckMessage` takes `msg` (required), `hint` (optional fix advice), `obj` (optional associated object), and `id` (optional stable identifier). The recommended `id` format is `{category}.{letter}{number}` — `letter` is `E` or `W`. Always provide an `id`, since `SILENCED_SYSTEM_CHECKS` references it.

Checks live in `<app>/checks.py` by convention. Django doesn't auto-discover; explicit registration in `AppConfig.ready()` is preferred over the `@register` decorator, because the decorator form relies on the checks module being imported as a side effect, which linters may flag as unused — silently disabling the checks if the import is removed. The book's two example checks: a warning when `settings.DEBUG and not sys.flags.dev_mode`, and an error when a model class name ends with `"s"` (singular names like `Book` read better with the ORM than `Books.objects.all()`). The model-name check excludes a `SAFE_MODEL_NAMES = {"PrintingPress"}` allow-list; a local allow-list is needed because `SILENCED_SYSTEM_CHECKS` silences a check ID *globally* across every instance, and Django ticket #26472 (which would add a per-instance silence mechanism) hasn't landed yet.

Custom checks return no errors in normal conditions, so without tests there's no ongoing assurance they still detect their target. Use Django's `SimpleTestCase` (no DB access, faster). Mock read-only attributes like `sys.flags.dev_mode` via `mock.patch.object(sys, "flags", SimpleNamespace(dev_mode=True))` — replacing the whole `sys.flags` object with one that has the required attribute. For model-name-style checks, decorate the test class with `@isolate_apps("example", attr_name="apps")` to give the tests a temporary, standalone app registry containing only the named app — local `class Sample(models.Model): pass` definitions inside test methods won't pollute the real registry.

Beyond standalone functions, Django piggybacks the check mechanism on framework classes: `Model` (classmethod), `Field` (instance), `Manager` (instance), `ModelAdmin` (via `checks_class`), and database backend `DatabaseValidation` all support `check()` overrides that the framework auto-discovers. Putting a model-name check on an abstract `NameCheckedModel` that subclasses opt into limits enforcement to its descendants, simplifying staged rollout in larger projects compared to a project-wide registered function.

The chapter closes with **django-version-checks** (another Johnson package), which compares actual runtime versions of Python and the major databases against expected ranges. Coordinating versions across dev laptops, CI, staging, and production is fiddly because each environment uses different config systems; a system check fits because it's fast and runs everywhere identically. The `VERSION_CHECKS` setting maps a name to a PEP 440 specifier (`"python": "~=3.10.1"` allows patch upgrades but not minor jumps); a non-compliant environment errors as `(dvc.E003) The current version of Python (3.9.9) does not match the specified range (~=3.10.1).` The recommended upgrade workflow is three-step: widen the range to admit both old and new versions, upgrade environments, narrow to the new version. Branching on `if DEBUG:` allows phased rollouts where developers see the new version requirement first, with time to flush out incompatibilities before staging and production follow.

## 10. Key takeaways

- **DX is a kaizen practice.** Continuous improvement to tools, conventions, and process — treating every bug or friction as feedback — is the book's organizing idea.
- **Default tools, well-configured.** Stick to venv + Pip + IPython + pre-commit + the standard formatter/linter stack rather than chasing newer alternatives. Universality beats novelty when team familiarity matters.
- **Pin, don't freeze.** `pip-compile` with separate `requirements.in` and `requirements.txt` files makes top-level versus transitive dependencies explicit, keeps annotations to explain why each package is included, and works the way npm and Cargo do.
- **Centralize configuration, but parametrize via env vars.** One settings file driven by `os.environ` (with a local `.env` for dev) beats per-environment files; a `.env-example` is the contract.
- **Put automated quality on the commit boundary.** EditorConfig + pre-commit + Black + isort + Flake8 + pyupgrade + django-upgrade together prevent whole categories of mistakes from reaching the repo. Modernizers run before formatters; linters run last.
- **The dev server is a feedback machine.** django-debug-toolbar makes invisible costs (especially N+1 queries) visible; django-browser-reload removes the manual refresh; Rich makes terminal output readable.
- **Prefer code over data for fixtures.** A `seed_database` management command plus Factory Boy is more maintainable than SQL backups or fixture files, and easier to test.
- **Treat migrations as a contract.** Test for pending migrations as a unit test, and use `django-linear-migrations` to refuse merge migrations rather than tolerate them.
- **System checks are the right place for environment invariants.** They run early, batch-report every issue, and inspect live runtime objects in ways linters can't. `django-version-checks` is the reference example.
- **Build small custom tools when needed.** Pre-commit's `language: fail`, `language: pygrep`, and `language: script`, plus local Flake8 AST plugins, cover the gap between off-the-shelf checks and bespoke needs without forcing a packaging step.
- **Production safety belongs in setup, not in vigilance.** `django-read-only` makes destructive shell mistakes structurally impossible; `DJANGO_READ_ONLY=1` exported from the production shell's rc file means the protection is on by default.
