## 1. pre-commit's virtual languages

pre-commit ships two "virtual languages" that let you write small custom hooks inline in `.pre-commit-config.yaml` without packaging a separate tool. Both go under `repo: local`, the special declaration meaning "this hook is defined in this repository".

### 1.1. "fail" hooks: block unwanted files

- **`language: fail`** — every file the hook matches automatically fails, with `entry` printed as the failure message. Pair with pre-commit's file/type filters to ban specific files from the repo.
- **Example: ban non-YAML fixtures** — after migrating fixtures from JSON to YAML, prevent old JSON fixtures from creeping back. Match all files inside any `fixtures/` directory (`files: /fixtures/`) and `exclude_types: [yaml]` so the hook fails on anything that *isn't* YAML.

  ```yaml
  - repo: local
    hooks:
    - id: check-fixture-types
      name: Check fixture types
      language: fail
      entry: Please convert non-YAML fixtures to YAML.
      files: /fixtures/
      exclude_types: [yaml]
  ```

- **`.gitignore` vs "fail" hooks** — use `.gitignore` for files that may exist locally but shouldn't be committed (`__pycache__`, JS bundles); use "fail" hooks for files that should not exist *at all* (legacy file types, naming patterns reserved by other tools like pytest's changelog files).

### 1.2. "pygrep" hooks: regex content search

- **`language: pygrep`** — fails files whose content matches a regex (Python's `re` module). Named after `grep` but cross-platform — vanilla `grep` differs by OS and isn't always installed.
- **Example: forbid `<form novalidate>`** — `novalidate` disables HTML5 client-side validation, useful when testing server-side validation but easy to forget to remove. A pygrep hook on `types: [html]` with `entry: novalidate` catches it before commit. (Don't try to parse HTML with regex generally; this works because "novalidate" wouldn't appear elsewhere.)
- **`description`** — optional human note for readers of `.pre-commit-config.yaml`.
- **More pygrep examples** — the `pre-commit/pygrep-hooks` repository (general-purpose Python checks), pytest's `\bpy\.path\.local` deprecation hook (with `exclude` shrinking over time as files are migrated), and the book's own `'^# Generated by Django'` hook to prevent dated comments in example migrations.

## 2. Custom Flake8 plugins

When pygrep is too crude and an existing Flake8 plugin doesn't exist, write a local plugin. Local plugins skip the Python packaging dance — start local, package later if you want to share.

### 2.1. The Abstract Syntax Tree

- **AST** — Python's parser output, stripped of formatting and comments, that Flake8 plugins receive. Easier to search than raw text.
- **Explore with `ast.parse()` + `ast.dump(..., indent=2)`** — `ast.parse('1 + 1')` returns a `Module` whose `body` holds an `Expr` whose `value` is a `BinOp(left=Constant(1), op=Add(), right=Constant(1))`. `print("Hello World")` yields `Expr(value=Call(func=Name(id='print'), args=[Constant('Hello World')]))`.
- **`ast.walk(node)`** — generator over every node in a tree. Combined with `isinstance(node, ast.X)` checks, this is the workhorse for finding "naughty" patterns.

### 2.2. Plugin structure

A Flake8 AST plugin is a class with:

| Member | Purpose |
|--------|---------|
| `name` (str) | Plugin name (display) |
| `version` (str) | `"unversioned"` is fine for local plugins |
| `__init__(self, tree)` | Receives the parsed `ast.Module` for the file |
| `run(self)` generator | Yields one tuple per error: `(lineno, col_offset, "CODE message", type(self))`. `lineno` is 1-based; `col_offset` is 0-based; both come from AST node attributes |

### 2.3. Wiring a local plugin

- **Example: ban `lambda`** — Guido van Rossum once proposed removing `lambda` from Python 3, so banning it isn't unreasonable. The plugin walks the tree and yields `L001 'lambda' is not allowed.` for every `ast.Lambda` node found.

  ```python
  class LambdaChecker:
      name = "No Lambdas"
      version = "unversioned"
      def __init__(self, tree):
          self.tree = tree
      def run(self):
          for node in ast.walk(self.tree):
              if isinstance(node, ast.Lambda):
                  yield (node.lineno, node.col_offset,
                         "L001 'lambda' is not allowed.", type(self))
  ```

- **Register in `setup.cfg`** — `[flake8:local-plugins]` with `extension = L = example.ext.flake8:LambdaChecker` (module:class, the same format as `pkgutil.resolve_name()`) and `paths = .` so Flake8 can import from the repo root. Codes starting with the `L` prefix are auto-selected unless `select` is explicitly configured.
- **Workaround for `lambda`** — the chapter notes `functools.partial` often replaces `lambda` cleanly (e.g., `partial(render, template_name=template_name)` instead of `lambda request: render(request, template_name)`).

### 2.4. Testing the plugin

- **Important because plugins succeed silently** — if your codebase already passes, the plugin's error path never runs in production; tests are the only assurance it still works.
- **Pattern with Django's `SimpleTestCase`** — wrap parsing + checker construction + collection of yielded errors in a `check()` helper using `textwrap.dedent()` for tidy multi-line examples. Cover a no-error case (`1 + 1`), a simple positive case (`lambda: 1` → one error at `(1, 0)`), and a nested case (lambda inside a function → error with the inner `lineno`/`col_offset`). Use `SimpleTestCase` because Flake8 doesn't run inside Django and can't touch the database.

## 3. Standalone custom tools

When even Flake8 isn't the right host, write a small standalone Python script. Place it in `scripts/` or `bin/` (not inside your project package, since it's not part of the importable API).

### 3.1. What pre-commit expects of tools

- Accept one or more filenames as positional arguments
- Return exit code 0 on success, non-zero on failure (and non-zero even when failures were *auto-fixed*, so the output gets shown)
- Print details to stdout/stderr
- Modify files in place (pre-commit detects changes via Git)

### 3.2. Example: rewrite Django doc URLs to /stable/

- **Problem** — Django doc URLs include the version (`docs.djangoproject.com/en/5.0/...`); rewriting them to `/stable/` keeps links evergreen as the project upgrades.
- **Tool design** — a *fixer* (auto-rewrites) rather than a linter (which would require manual edits). `argparse` with `nargs="+"` and `type=argparse.FileType("r+", encoding="utf-8")` opens each file for read/write automatically. `re.subn(r"https://docs\.djangoproject\.com/en/\d\.\d/", "https://docs.djangoproject.com/en/stable/", text)` returns the new text and a substitution count; >0 substitutions means the file changed.
- **Writing back to a re-opened file** — `file.seek(0)` to rewind, `file.write(new_text)`, then `file.truncate()` to declare the new end-of-file (without it, shrinking files leave trailing garbage).
- **`exit_code` starts at 0, becomes 1 on any change** — matches pre-commit's "fail when fixes were applied" expectation.

### 3.3. Wiring with `language: script`

- **`language: script` invokes any executable** — `entry: scripts/fix_django_links.py` is the path relative to the repo root. The OS reads the shebang (`#!/usr/bin/env python`) — pre-commit doesn't care which language the tool is written in.
- **Make the file executable** — `chmod +x scripts/fix_django_links.py` on Unix.
- **Scope with `types_or`** — list relevant file types (`markdown`, `python`, `rst`) so the tool only sees files whose content might contain Django doc links.

### 3.4. Testing with pytest

- **pytest beats unittest for tool tests** — its fixtures make external-tool testing concise.
- **Two key fixtures** — `tmp_path` provides a temporary `pathlib.Path` directory, auto-cleaned after the test; `capsys` captures stdout/stderr so assertions can verify the tool's output.
- **Arrange-Act-Assert pattern** — write an example file under `tmp_path`, call `main([str(example)])`, then assert exit code is 1 (file changed), `capsys.readouterr()` matches expected output, and the file content was rewritten as expected.
