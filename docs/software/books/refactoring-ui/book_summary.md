## 1. Thesis and approach

Refactoring UI is a practical book for developers — people who can build interfaces but feel blocked by the assumption that good design takes artistic talent. Schoger and Wathan's central claim is that polished UI is not the result of taste so much as the result of many small, concrete, rule-based decisions. Every chapter converts something that looks like intuition — when a button should be big, what shade of grey to use, whether an icon belongs next to a heading — into a decision procedure.

Two themes run through every rule. The first is **work in small loops**. Don't try to design "the app" in the abstract; start with a single feature, ship a basic version, let the real interface expose problems, then iterate. The book opens by telling the reader not to start with the navigation bar, not to start with color, not to design every feature up front, and not to imply functionality that isn't ready to build. A comment system with no attachments is shippable; a comment system blocked on attachments is not. Schoger and Wathan push for low-fidelity sketches (Jason Fried of Basecamp designs on paper with a thick Sharpie so the tool itself prevents obsessing over detail) and grayscale mockups, because stripping away color forces spacing, contrast, and size to carry the hierarchy — which produces a stronger interface that is easier to enhance with color later.

The second theme is **constrain your choices with systems**. Unlimited options produce paralysis. "Should this be 12px or 13px? Should this avatar be 24px or 25px?" With no constraints there is always more than one right answer, so decisions are torture. The fix is to define a small set of options in advance — a type scale, a shadow system, 8–10 greys, 5–10 primary shades — and pick from that set instead of from a limitless pool. When you only have to choose between 12/16/24/32px, guess at one, compare to the values on either side, and two options will look obviously wrong. The more systems you have in place, the faster you work and the less you second-guess. This philosophy reappears in every chapter: the type scale replaces ad-hoc font sizes, the color palette replaces `lighten()`/`darken()` calls, the spacing scale replaces pixel-by-pixel margin fiddling, and the elevation system replaces one-off shadow tweaks.

The book calls this mindset "choosing a personality" at the small scale — deliberately picking fonts, colors, border radius, and language that communicate something specific (secure/professional vs. fun/playful, elegant/classic vs. plain) — and at the large scale it is simply: make decisions once, in a system, and reuse them.

## 2. Hierarchy is the engine

If there is one idea that carries the book, it is **visual hierarchy** — how important elements appear relative to one another. When everything on a page shouts equally, the result is a chaotic wall of content where nothing matters. Deliberately de-emphasizing secondary and tertiary information is what makes the important bits pop, without changing color, type, or layout.

A beginner's instinct is to create hierarchy with font size alone, which produces primary content that is too large and secondary content that is too small. The book's alternative is to lean on **font weight and color** instead. A primary headline can stay at a reasonable size if it is heavier and darker; a date next to it can stay readable if it is lighter grey with a normal weight. The authors recommend two or three colors for text hierarchy — a dark color for primary content, a grey for secondary, a lighter grey for tertiary — and two font weights: a normal weight (400 or 500) for most text and a heavier weight (600 or 700) for emphasis. Weights under 400 are a trap: they can work for large headings but become unreadable at smaller sizes, so reach for a lighter color or a smaller font size instead of a lighter weight.

Grey text works against a white background because it has *less contrast* with the background, not because it is grey — so the same grey on a colored background looks dirty rather than de-emphasized. White at reduced opacity is worse; it washes out and lets background images bleed through. The fix is to hand-pick a tinted color with the same hue as the background and adjust saturation and lightness until it reads as supporting text.

The most counterintuitive hierarchy move is **emphasize by de-emphasizing**. When a main element won't stand out no matter what you do, soften the elements competing with it instead. An active nav item "pops" once the inactive items are given a softer color; a sidebar that is fighting the main content settles back once its background color is dropped entirely.

Closely related: **labels are a last resort**. Giving every piece of data a "Label:" prefix forces equal emphasis on all of it and kills hierarchy. Most labels are redundant — `janedoe@example.com` is obviously an email, `(555) 765-4321` is a phone number, `$19.99` is a price, and "Customer Support" under a name in an employee directory reads as a department without being labeled. When the data needs context, merge label and value into one unit: "12 left in stock" instead of "In stock: 12"; "3 bedrooms" instead of "Bedrooms: 3". When labels are genuinely necessary — a dashboard with many similar fields — treat them as secondary: smaller size, lower contrast, lighter weight. The exception is information-dense pages like technical specs, where users scan for the label ("depth"), not the value ("7.6mm"), and the label should be darker than the value.

Two more rules complete the hierarchy chapter. **Visual hierarchy and document hierarchy are separate concerns.** Browsers default `h1` to huge and `h6` to small, which makes sense for articles but misleads in app UIs. A "Manage Account" page title is legitimately an `h1` but doesn't need to be enormous; a section title is often supporting content and sometimes should be visually hidden (kept in markup for accessibility) when the content speaks for itself. Pick tags for semantics and style them for hierarchy.

**Balance weight and contrast.** Bold text feels heavy because it covers more pixels per unit of background — a property that generalizes to any UI element. When a solid icon next to text dominates, you can't make the icon less bold, but you can lower its contrast with a softer color to rebalance. The reverse also works: a thin 1px border that is too subtle in a soft color but too harsh when darkened can simply be thickened — more weight restores emphasis without losing the softer look.

Finally, **semantics are secondary** when styling action buttons. Every page has one true primary action, a couple of secondary actions, and a few tertiary actions, and the styling should match that pyramid — solid high-contrast for primary, outline or lower-contrast for secondary, link-styled for tertiary. A destructive action is not automatically primary just because it is high-severity. If "delete account" is not what the user came for, give it a secondary or tertiary treatment on the page and save the big, red, bold styling for the confirmation step, where destroying genuinely *is* the primary action.

## 3. Layout and spacing

Layout decisions suffer from the same systematic pathology as text decisions: too many options and the wrong default. Schoger and Wathan flip both.

First, **white space should be removed, not added**. The default web habit is to add margin only until something stops looking bad, which gives every element the minimum breathing room rather than enough to look great. Start with way too much space and remove until you're happy. What feels excessive when zoomed in on a single element usually settles into the right amount in context. Dense UIs — dashboards that must show a lot of data on one screen — are a legitimate exception, but the key is making compactness a deliberate choice rather than a default.

Second, **establish a spacing and sizing system**. A linear "multiples of 4" scale doesn't help you choose between 120px and 125px because the *relative* difference between adjacent values matters more than the absolute one. At the small end, a few pixels is a lot — jumping an icon from 12px to 16px is a 33% increase. At the large end, a few pixels is nothing — growing a card from 500px to 520px is 4%, eight times less significant. The authors' rule of thumb: make sure no two adjacent values on your scale are closer than about 25%, so every step produces a visibly different result. Start from a sensible base like 16px (it divides cleanly and is the default browser font size) and build the scale with factors and multiples.

Third, **you don't have to fill the whole screen**. Modern 1200–1400px canvases tempt you to use every pixel, but if an interface only needs 600px, use 600px. Extra space around the edges never hurts. Just because the nav is full-width doesn't mean every section under it must be. When something works best at a narrow width but feels unbalanced on a wide canvas, split it into columns rather than stretching it — break a form's supporting text into a second column so the form itself stays at its optimal width. If a small interface is hard to design on a big canvas, shrink the canvas; for responsive apps, design mobile-first and adjust when you move to larger screens.

Fourth, **grids are overrated**. A 12-column grid gives elements fluid percentage widths (each column is 8.33%), which is only useful when you actually want the element to scale. A sidebar given 3 of 12 columns grows too wide on large screens (wasting space the content could use) and shrinks below its minimum usable width on narrow screens. Better: fix the sidebar to its content and let the main area flex. A login card given 6 of 12 columns can end up *wider* on a medium screen (8 columns of a smaller canvas) than on a large screen, which is silly — give it a `max-width` instead and only shrink when the viewport forces it.

Fifth, **relative sizing doesn't scale**. If 18px body and 45px headline feel right on desktop, encoding the headline as `2.5em` is tempting but misleading: on small screens where body text shrinks to 14px, `2.5em` becomes 35px — far too big. Large elements should shrink faster than small ones as screens get smaller; a mobile headline might be 20–24px, a different *ratio* to the body entirely. The same applies inside components — tying a button's padding to its font size makes a large button feel like a zoomed version of a small one, when padding should get disproportionately more generous at larger sizes and tighter at smaller sizes.

Sixth, **avoid ambiguous spacing**. When groups of elements aren't separated by a border or background, spacing is what tells the user which elements belong together. A form label followed by its input with *equal* spacing above and below the input looks like two independent items rather than a labeled pair. The rule of thumb: whenever you're using spacing to group elements, make sure there is more space around the group than within it. Undersized space above article section headings attaches them to the wrong paragraph. Bullet spacing equal to the line-height of a single bullet blurs items together. The failure mode is identical in horizontal layouts.

## 4. Designing text

Typography is the layer of a UI most beginner designers outsource to Google Fonts without a plan. The chapter on text is largely about building another system — the **type scale** — and a few heuristics for picking fonts.

The type scale replaces the default failure mode of using nearly every pixel value between 10px and 24px. A useful scale is non-linear — smaller jumps at the bottom (12 → 14 → 16 is fine) and larger jumps at the top so you aren't agonizing over 46px vs. 48px. Modular scales based on mathematical ratios (4:5, 2:3, the golden ratio 1:1.618) produce fractional pixel values like 31.25px and 39.063px that browsers round inconsistently, so the authors recommend hand-crafted scales. Avoid defining the scale in `em` units — `em` is relative to the current font size, so nested elements compute to sizes outside the system (an element styled `.875em` inside a parent at `1.25em` on a 16px base lands at 17.5px, not a scale value). Stick to `px` or `rem`.

Picking a good font without years of typographic experience is shortcut-able with a handful of heuristics. **Play it safe** — a neutral sans-serif like Helvetica, or the system font stack (`-apple-system, Segoe UI, Roboto, Noto Sans, Ubuntu, Cantarell, Helvetica Neue`), is an unambitious but safe default that users already recognize. **Ignore typefaces with less than five weights** — families with many weights tend to be built with more care; filtering Google Fonts to 10+ styles cuts 85% of the catalog and leaves fewer than 50 sans-serifs. **Optimize for legibility** — fonts meant for headlines have tight letter-spacing and short x-heights; fonts for body text have wider spacing and taller lowercase letters. Don't pick a condensed headline face for body UI. **Trust popularity** — sort font directories by the number of users and borrow from sites you admire, inspecting their type choices.

Readable paragraph text comes down to two numbers. **Line length**: 45 to 75 characters, which on the web corresponds to roughly 20–35em of width. If paragraphs sit alongside wider components, constrain only the text column rather than stretching prose to match. **Line-height**: not a fixed 1.5 but proportional to line length — the longer the lines, the bigger the horizontal eye jump to the next line, so wider content needs more line-height (up to 2). Large headlines can run at a line-height of 1 because each line is short and the eye has no trouble finding the next. Line-height is inversely proportional to font size: small text needs more spacing, big text needs less.

When two different font sizes share a line — a large card title next to smaller actions — align them to the baseline, not the vertical center. Centering offsets their baselines, and the baseline is a reference the eye already perceives. For body text, left-align everything in English and most languages; center alignment is fine for short headlines or independent blocks but not for paragraphs longer than two or three lines. If you must justify text (for a print or magazine feel), enable hyphenation to close the word gaps justification creates on the web. Right-align numbers in tables so decimals line up vertically.

Links need less decoration than browser defaults suggest. In a paragraph of body text a colored, underlined link "pops" appropriately, but in interfaces where almost everything is a link, that default becomes overbearing. Emphasize most links subtly — a heavier font weight or slightly darker color is enough — and let ancillary links skip default emphasis entirely, showing their underline only on hover. They remain discoverable without competing with primary actions.

Trust the typeface designer and generally leave letter-spacing alone. Two exceptions: **tighten headlines** when you're using a body-optimized face for titles (Open Sans, say, in place of a purpose-built headline font like Oswald), and **increase letter-spacing for all-caps text** because uniform-height capital letters lose the shape distinctions (ascenders like b/f/t, descenders like y/g/p) that help the eye separate words in sentence case.

## 5. Working with color

Color is the area where systems matter most, and where the book's rules depart most sharply from common practice.

First, **ditch hex for HSL**. Hex and RGB hide visual relationships — two colors that look closely related to the eye can look nothing alike in code. HSL represents a color using three intuitive attributes that match human perception: **hue** (position on the color wheel, measured in degrees — 0° red, 120° green, 240° blue), **saturation** (0% grey, 100% vivid), and **lightness** (0% black, 100% white, 50% a pure color at that hue). Nudging an HSL value produces a predictable visual change. Don't confuse HSL with HSB, which is more common in design tools but behaves differently — at 100% saturation, 100% HSB brightness equals 50% HSL lightness. Browsers only speak HSL, so use HSL on the web.

Second, **you need more colors than you think**. The palette-generator trap is picking five "perfect" colors and assuming that's a palette. Real UIs need far more. Schoger and Wathan break the palette into three categories. **Greys** cover almost everything — text, backgrounds, panels, form controls — and need 8 to 10 shades. True black looks unnatural, so start from a very dark grey and step up to white. **Primary colors** (one or two) carry the site's identity: Facebook blue, primary buttons, active nav. They need 5 to 10 shades — the ultralight ones for tinted alert backgrounds, the darker ones for colored text. **Accent colors** are eye-grabbers for new features and semantic signals — red for destructive confirmations, yellow for warnings, green for positive trends. They are used sparingly but still need multiple shades. Complex UIs can end up needing up to ten colors with 5 to 10 shades each, especially when color has to distinguish graph lines, calendar events, or tags.

Third, **define your shades up front**. Don't generate shades on the fly with CSS preprocessor functions like `lighten()` or `darken()`; that's how you end up with 35 near-identical blues scattered across a codebase. Choose a fixed palette and pick from it. Pick the base color first (the middle shade — would this color work as a button background?); then pick the darkest (typically for text) and lightest (typically for tinted backgrounds) by thinking about where they'll be used; then fill in the gaps. Nine shades is a useful number because it divides cleanly: 900 (darkest), 500 (base), 100 (lightest); then 700 and 300 as midpoints; then 800, 600, 400, 200 the same way. For greys, skip the base and start from the edges — darkest is the main text color, lightest is a subtle off-white background. It's not a science; trust your eyes and tweak once you start using the shades in real designs. But avoid extending the palette constantly — a palette you keep growing isn't really a system.

Fourth, **don't let lightness kill your saturation**. The same saturation value that looks vivid at 50% lightness looks washed out at 90%. To keep lighter and darker shades from feeling dull, increase saturation as lightness moves away from 50%. Every hue also has an inherent perceived brightness — the eye reads yellow as brighter than blue at the same HSL lightness. Instead of only adjusting lightness (which always pulls toward white or black and saps intensity), you can rotate the hue toward a naturally brighter or darker hue: to lighten, rotate toward 60°, 180°, or 300°; to darken, rotate toward 0°, 120°, or 240°. Keep rotation to 20–30° or the result reads as a different color rather than a lighter or darker version.

Fifth, **greys don't have to be grey**. Pure grey has 0% saturation, but most greys in polished UIs are lightly saturated, which gives them a temperature. Like warm-white vs. cool-white light bulbs, greys tinted toward blue feel cool, and greys tinted toward yellow or orange feel warm. Apply the temperature across the whole scale and remember to raise saturation on the lightest and darkest greys too, for the same reason as primary colors.

Sixth, **accessibility without ugliness**. The Web Content Accessibility Guidelines require a contrast ratio of at least 4.5:1 for normal text and 3:1 for larger text. Dark-on-light hits this easily; colored backgrounds make it harder. A common failure mode is white text on a colored background that needs the color pushed so dark to reach 4.5:1 that the element over-emphasizes itself. The fix is often to flip the contrast — dark colored text on a light colored background of the same hue — so the color still supports the text but the element stops screaming. When placing colored text inside a dark colored panel, rotating the hue toward naturally brighter hues (cyan, magenta, yellow) can reach the required contrast while keeping the text colorful, without going nearly white.

And, relatedly, **don't rely on color alone**. Users with color blindness can't read information encoded only in hue. Add icons to red/green metric cards; for charts with many series, rely on contrast rather than distinct hues — colorblind users distinguish light-vs-dark far more easily than two colors. Color should always support something the design is already communicating, never carry the message by itself.

## 6. Creating depth

Flat-ish UIs still communicate depth — just with more subtlety than skeuomorphism did. The rules come down to one physical intuition: **light comes from above**. A real-world raised panel shows a lighter top edge (angled toward the sky) and a darker bottom edge (angled away); an inset panel reverses it. Mimic this and the brain reads depth automatically.

Because users look slightly *downward* at their screens, you generally only show one edge at a time. For **raised** elements like buttons, reveal the top edge with a subtle highlight (a top border or inset shadow with a slight positive vertical offset, slightly lighter than the face) and add a small dark cast shadow with a slight positive vertical offset beneath the element, sharp-ish (a couple of pixels of blur, not more). For **inset** elements like wells, text inputs, and checkboxes, reveal the bottom lip with a subtle lighter border at the bottom and add a small dark inset shadow at the top, as if the upper lip is blocking light. Hand-pick the highlight color rather than overlaying semi-transparent white, which sucks saturation out of the underlying color. And keep shadows sharp and subtle — real-world references like the shadow under a wall outlet or window frame have sharp edges, not giant soft blurs.

**Shadows convey elevation**, not decoration — they position elements on a virtual z-axis. A small, tight-blur shadow puts a button just off the background; a medium shadow lifts a dropdown above the UI; a large, softer shadow brings a modal close to the user's face. Build an elevation system with about five options, pick the smallest and largest first, then fill the middle with roughly linear increments. Think in *z-axis* instead of fiddling with blur and offset — assign the matching system shadow once you decide where the element should sit. Shadows can also act as interaction cues: adding a shadow on click makes an element pop forward (and signals it can be dragged), while switching to a smaller shadow — or removing it — makes a button feel pressed into the page.

Nice shadows on real sites are often two shadows stacked. The **first** is larger, softer, with a considerable vertical offset and large blur — simulating the shadow cast behind an object. The **second** is tighter, darker, with a smaller offset and blur — simulating the shadowed area directly under the object where ambient light can't reach. This gives more control than a single shadow: the diffuse shadow stays subtle while the tight one keeps the edges near the element crisp. As elevation increases, the tight ambient-occlusion shadow fades; make it distinct at the lowest elevation and almost invisible at the highest.

"Flat design" usually means no shadows or gradients, but the strongest flat designs still convey depth. **Depth with color**: among shades of the same color, lighter reads as closer and darker reads as further away. An element lighter than its background feels raised; darker feels inset like a well. And **solid shadows** — short, vertically offset, with no blur at all — can lift a card or button off the page without breaking the flat aesthetic.

Finally, **overlap elements to create layers**. Letting elements break out of their containers makes a design feel multi-layered rather than stacked. A card that straddles two background colors; an element taller than its parent that overflows on both sides; a carousel control that pokes past the image edge. When overlapping images clash, give each a border matching the background color so a small gap always separates them — the layered look stays, the clashing goes.

## 7. Working with images

Photos can elevate or sink a design, and most of the chapter is about getting them right rather than letting them drag the design down.

**Use good photos.** No amount of layout, typography, or color saves a design whose imagery is weak. For project-specific imagery, hire a professional — good photos are about lighting, composition, and color, not about owning an expensive camera. For generic needs, paid stock libraries and free sources like Unsplash are fine. The failure mode is designing with placeholder images and "fixing it later" with smartphone snaps at the end. Decide on real imagery up front.

**Text needs consistent contrast.** Photos contain both very light and very dark areas, so a single text color will get lost somewhere. The fix isn't the text but the image — reduce the image's dynamic range so contrast against the text is consistent everywhere. Four techniques: a semi-transparent overlay (black tones down light areas for light text; white brightens dark areas for dark text) — simple but darkens the whole image; lowering the image's contrast and compensating brightness — more control, but requires editing the source; colorizing the image (lower contrast, desaturate, apply a solid fill with the "multiply" blend mode), which also ties the image to brand colors; or a text shadow with a large blur radius and no offset — reads like a glow and boosts contrast only where the text actually is, best combined with a small contrast reduction.

**Everything has an intended size.** Bitmap images get fuzzy when scaled up, but vectors and screenshots have their own failure modes. Don't scale up vector icons — a 16px icon at 3x its intended size looks disproportionately chunky and lacks detail. When small icons are all you have and you need to fill a larger space, enclose them in a shape with a background color so the icon stays near its intended size while the container occupies the space. Don't scale down screenshots either — a full-size screenshot shrunk 70% to fit crams too much detail into too little space; a 16px font in the app becomes 4px in the screenshot. Options: shoot at a smaller layout (tablet), show only a partial screenshot of a meaningful slice, or redraw a stripped-down UI with small text replaced by simple lines. Don't scale icons down either — they go choppy and fuzzy. Favicons are the extreme case: a 128px logo crunched into 16px turns to mush because the browser has to invent how to render that detail. Redraw a simplified version at the target size and control the compromises yourself.

**Beware user-uploaded content.** You can't fine-tune the content, but you can stop it from wrecking the layout. Control shape and size: display uploads at a fixed aspect ratio by centering them inside fixed containers and cropping whatever doesn't fit (CSS: background image with `background-size: cover`). And prevent background bleed — when an uploaded image's background matches the UI background, the image loses its shape. A border fights the image's colors; a subtle inner box shadow defines the edge without clashing and mostly goes unnoticed. A semi-transparent inner border is a viable alternative.

## 8. Finishing touches

The final chapter is about the polish that separates a competent-but-bland design from one that feels "designed." Schoger and Wathan's advice here is low-effort: you don't have to be a professional illustrator to add personality.

**Supercharge the defaults.** A lot of polish comes from upgrading the elements already on the page rather than adding new UI. Replace bulleted list markers with icons (checkmarks, arrows, or content-specific ones like a padlock for security features). Promote testimonial quotes visually by enlarging and recoloring the quote marks. Style links beyond the browser default — a color and weight change works, or a thick colorful underline that partially overlaps the text. Customize checkboxes and radio buttons with a brand color for the selected state.

**Add color with accent borders.** A colored rectangle across the top of a card, under an active nav item, along the side of an alert, beneath a headline, or across the entire layout requires no graphic-design talent but makes a UI feel deliberate.

**Decorate your backgrounds.** When everything else is right but a design still feels plain, the background is usually why. Three low-effort moves: change the background color (for a whole section or a single panel; a slight gradient with the two hues less than ~30° apart adds energy without chaos); use a repeating subtle pattern across the whole background or along one edge; or add a simple shape or illustration. In all three, keep contrast between background and pattern low so it doesn't fight the content.

**Don't overlook empty states.** If a feature depends on user-generated content, the empty state is the user's *first* interaction with the feature — and a blank screen kills enthusiasm for a feature you spent weeks building. Use an image or illustration plus a clear call-to-action that pushes the user toward the next step. Hide supporting UI (tabs, filters) that has nothing to act on. Treat empty states as an opportunity to be interesting.

**Use fewer borders.** Borders create separation but too many make an interface feel busy. Before reaching for a border, try a box shadow (more subtle when the element's background differs from the page), try two different background colors on adjacent elements (often enough distinction on its own), or just add more spacing. Spacing is the most direct form of separation and adds no new UI.

**Think outside the box.** The preconceived shape of a component is not a rule. A "dropdown" doesn't have to be a white box with a drop shadow and a stack of links — it's a floating container and can hold sections, multiple columns, supporting text, or icons. Table cells can hold images or color, not just plain text; a column that doesn't need to be sortable can be combined with a related column to introduce hierarchy. Radio buttons can become selectable cards when the choice is a central part of the UI — a stack of labels with little circles is the least interesting option available. Don't let habit about "how a component is supposed to look" cap the quality of the design.

## 9. Leveling up

The book's closing chapter is short and blunt: design skill compounds through deliberate study.

Every time you come across a design you like, ask the same question: *did the designer do anything here that I never would have thought to do?* The unintuitive choices — inverting the background color on a datepicker, placing a button inside a text input instead of outside it, using two different font colors in a single headline — are where new ideas come from. Paying deliberate attention to small deviations from the default is how you build a personal library of tricks.

The exercise that accelerates this most is **rebuild your favorite interfaces from scratch, without peeking at DevTools**. The act of trying to match the original forces you to notice every detail that makes it polished — reducing line height on headings, adding letter-spacing to uppercase text, combining multiple shadows — details you'd never read in a tutorial but that you uncover by chasing down the gap between your version and the original. The practice compounds: studying work that inspires you, carefully, keeps adding tricks to your toolbelt for years.

## 10. Key takeaways

- **Design is a set of rule-based decisions**, not taste. Every instinct in this book has a procedure behind it.
- **Start with features, not layouts.** An app is a collection of features; you don't have the information to design the shell until you've designed some actual functionality.
- **Work in cycles.** Design a simple version, ship it, iterate in the real interface where imagination can't. Don't imply functionality you aren't ready to build.
- **Constrain your choices with systems** — type scale, spacing scale, color palette, shadow/elevation system. Do the hard work once, not every time you build UI.
- **Hierarchy is the engine.** Use font weight and color before font size. De-emphasize the competition instead of amplifying the target. Separate visual hierarchy from document hierarchy.
- **Labels are a last resort.** Format and context usually remove the need for them; when they're necessary, treat them as secondary.
- **White space is removed, not added.** Start with too much, strip back.
- **Don't fill the whole screen** just because it's there. Use the space the content needs.
- **Use HSL, not hex.** You need 8–10 greys, 5–10 shades of each primary, and several accent shades — far more than a five-color palette. Define them up front.
- **Saturation fades at extreme lightness.** Raise saturation — or rotate the hue — to keep lighter and darker shades vivid.
- **Light comes from above.** Shadows position elements on a z-axis. Build an elevation system and pick from it.
- **Contrast generalizes beyond text.** Bold, solid icons, and heavy borders all carry visual weight; compensate with softer colors when they dominate.
- **Polish comes from upgrading defaults, not adding UI** — custom bullets, accent borders, decorated backgrounds, considered empty states, fewer borders, unconventional component shapes.
- **Leveling up is ongoing**. Rebuild interfaces you admire; ask what the designer did that you wouldn't have.
