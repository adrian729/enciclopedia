## 1. Thesis: Patterns to Democratize Distributed Systems

Modern always-on applications and APIs face availability, reliability, and viral-growth scaling demands once reserved for a handful of mission-critical systems. As a result, almost every application built today — whether a consumer mobile app or a backend payments service — needs to be a distributed system. But building reliable distributed systems is hard. The engineering skills required exceed those needed for single-machine applications, and most distributed systems today are still bespoke, one-off solutions. Burns argues that this state of affairs resembles software development before the rise of object-oriented programming: ad hoc and dependent on rare expertise.

The book's central claim is that the combination of containers, container orchestrators, and a shared catalog of distributed-system patterns can change this — much as Knuth's algorithmic formalization (1962) crystallized procedural programming, and the Gang of Four's *Design Patterns* did for object-oriented programming. A **pattern**, in Burns's sense, is a general blueprint for organizing distributed systems independent of any specific technology or application; not a recipe for installing a NoSQL database or composing a MEAN stack. Patterns offer three values: they let practitioners stand on the shoulders of giants and learn from others' mistakes; they provide a shared vocabulary that ends arguments of "violent agreement" between people using different names for the same idea; and — when implemented as container images with HTTP-based interfaces — they yield reusable components that can be shared across programming languages and broadly battle-tested. The remainder of the book catalogs these patterns across three settings: single-node composition, long-running serving, and batch computation.

## 2. Single-Node Composition

The first family of patterns combines two or more containers on a single machine, coscheduled into an atomic **container group** such as a Kubernetes `pod`. Containers in such a group share resources — parts of the filesystem, hostname, network, and many other namespaces — which makes them tightly cooperative without requiring source-level integration. Burns identifies three distinct roles a second container can play: sidecar, ambassador, and adapter.

### 2.1. The Sidecar Pattern

The **sidecar pattern** pairs an *application container* (which holds the core logic) with a *sidecar container* that augments or extends it, often without the application container's knowledge. A canonical example: a legacy web service that only speaks HTTP and whose old build system no longer functions. Rather than reviving the build system or porting the code, the team binds the legacy app to localhost (127.0.0.1) and adds an `nginx` sidecar in the same network namespace. The sidecar terminates HTTPS on the pod's external IP and proxies plaintext traffic over the loopback adapter — modernizing the legacy app without touching its source.

Sidecars are useful far beyond legacy adaptation. A configuration-manager sidecar can poll a cloud config API and write changes to a shared file, signaling the application via file watches, `SIGHUP`, or — as a last resort — `SIGKILL` plus an orchestrator restart, adapting any config-file-driven app to dynamic configuration. A `topz` sidecar can share an application's PID namespace and expose a uniform `/topz` web UI of running processes, replacing per-language plugin libraries with a single container the orchestrator can attach to every workload. Sidecars can even compose application behavior: a Node.js server using `nodemon` plus a sidecar running `while true; do git pull; sleep 10; done` is a minimal Git-push-to-deploy PaaS.

### 2.2. The Ambassador Pattern

The **ambassador pattern** also pairs the application with a second container, but the ambassador's role is specifically to broker all interactions with the outside world. The application connects to a fixed localhost address, oblivious to the real topology, while the ambassador owns environment- and deployment-specific logic. This produces three concrete uses.

A **sharding ambassador proxy** lets unmodified single-backend client code talk to a sharded storage service. The ambassador exposes what looks like one backend on localhost but routes each request to the correct shard. Burns demonstrates this with a Kubernetes `StatefulSet` of three Redis shards fronted by `twemproxy` running as the ambassador. A **service broker ambassador** solves the portability problem when the same application must run across public clouds, private clouds, and on-prem datacenters where dependencies are provisioned differently. The ambassador performs *service discovery* (locating the appropriate dependency for the current environment) and brokers the connection, isolating environment logic from application logic. Finally, a **request-splitting ambassador** sends a fraction of traffic to an experimental version of a service. The book's hands-on example uses an `nginx` ambassador with a 9:1 weighted upstream and `ip_hash` to give each user a consistent experience while running a 10% experiment.

### 2.3. The Adapter Pattern

Where the ambassador modifies the application's *outbound* interface, the **adapter pattern** modifies its *inbound* interface so it conforms to a system-wide convention — for example, a uniform monitoring API or a convention that all logs go to `stdout`. Real applications mix in-house code, vendor binaries, and off-the-shelf components written in many languages with incompatible conventions; an adapter container lets the surrounding system see one consistent interface across all of them.

For monitoring, an adapter container transforms whatever interface the application natively offers into the format a single monitoring stack expects. Burns's hands-on example pairs the stock `redis` image with the open-source `oliver006/redis_exporter` adapter to make Redis Prometheus-monitorable with no custom code. For logging, an adapter redirects file-based logs to `stdout` and normalizes diverse formats into one structured representation; a `fluentd` adapter with the `fluent-plugin-redis-slowlog` plugin turns Redis's live-only `SLOWLOG` into persistent queryable logs. For health checks, an adapter container holds a representative query (e.g., a real MySQL test query) that the orchestrator probes for richer health information than a simple "process running" check.

A theme runs through these examples: when the application is owned, modifying it directly is fine; but when reusing a third-party container, deriving and maintaining a patched image is more expensive than running an adapter alongside it, and an external adapter can be shared and reused while a modified image cannot.

### 2.4. Cross-Cutting Discipline: Modularity and Reuse

Sidecars, ambassadors, and adapters all derive their value from being *reusable artifacts*. To make them so, Burns prescribes three disciplines. First, **parameterize** containers like functions: enumerate inputs that customize the generic image to a specific deployment (e.g., a certificate name and a localhost port for an SSL sidecar) and prefer environment variables to command-line arguments for delivering them. Second, **define and respect the container's API**, treating it as a microservice contract: parameters, outbound calls, and inbound HTTP/other interfaces all form the API. Breaking changes can be subtle — renaming `UPDATE_FREQUENCY` to `UPDATE_PERIOD` is obviously breaking, but so is changing the units of a value (seconds to strings like "10m"; or unitless integers from seconds to milliseconds), even if it doesn't error, because behavior silently changes. Third, **document with Dockerfile directives**: `EXPOSE` for ports with explanatory comments, `ENV` for default parameter values with usage notes, and `LABEL` using the Label Schema project's taxonomy so community tools can visualize, monitor, and use the image without bespoke configuration.

## 3. Long-Running Serving Patterns

The second family of patterns is for long-running services — the workloads people most often call "microservices." Three patterns handle the dominant cases: replication for stateless scale, sharding for state too large to fit on one machine, and scatter/gather for parallelizing the compute of a single request.

### 3.1. Replicated Load-Balanced Services

The **replicated load-balanced service** is the simplest distributed pattern: a scalable number of identical servers behind a load balancer that distributes traffic round-robin or with session stickiness. Because every replica can serve every request, the pattern fits **stateless services** — those that require no saved state to operate, like static-content servers or middleware that aggregates responses.

Even the simplest service should run at least two replicas. A "three-nines" SLA (99.9% availability) allows only 1.4 minutes of downtime per day; with a single instance, every rollout must finish inside that budget, and continuous delivery (e.g., hourly pushes) leaves only 3.6 seconds per rollout. Two replicas behind a load balancer make rollouts and crashes invisible. To know when a replica is *ready* to receive traffic — distinct from being merely alive — applications expose a **readiness probe**, a dedicated URL the load balancer polls. The probe matters because containers may be alive but still loading plugins, connecting to a database, or downloading serving files (the book's hands-on example uses a NodeJS dictionary server that downloads ~8 MB of data before reporting ready).

The pattern extends naturally to **session tracked services** (a.k.a. session affinity), where all of one user's requests are pinned to the same replica — useful when per-user caching raises hit rate or when state carries between requests. The typical implementation hashes source and destination IP, but this breaks across NAT for external users, where cookie- or header-based application-level tracking is preferred. A **consistent hashing function** is essential here: it ensures that scaling the replica count up or down only remaps a small fraction of users instead of reshuffling almost everyone.

Knowing the protocol (typically HTTP) lets the load-balancing tier do more than blind TCP/IP forwarding. A **caching web proxy** like Varnish in front of stateless backends serves identical responses from memory, sparing the backend. Burns warns against deploying the cache as a sidecar next to each web server — every page would duplicate across replicas, shrinking the working set and lowering the **hit rate** (fraction served from cache). A separate caching tier with few large replicas (e.g., two with 5 GB each) outperforms many small ones (ten with 1 GB each), because web servers stay numerous and small to use multiple cores while caches stay few and fat to maximize unique pages held in memory. Caching also forces session affinity to move from IPs to cookies, since the backend now sees only the caches' IPs. The same edge layer hosts **rate limiting** (denial-of-service defense against both attacks and accidents — a misconfigured client, an SRE running a load test against production), with anonymous limits kept low to push users to log in and `429` returned with `X-RateLimit-Remaining`-style headers when limits are hit. **SSL termination** sits even further out: nginx terminates HTTPS in front of Varnish, with each layer using its own certificate so layers can roll out independently.

### 3.2. Sharded Services

A **sharded service** differs from a replicated one in that each replica — a *shard* — serves only a subset of requests. Each request first hits a **root** load-balancing node that inspects the request and routes it to the appropriate shard. Replicated services are typically stateless and scale for redundancy and request volume; sharded services are typically stateful and scale because the *state itself* is too large for one machine.

The motivation is clearest in caching. With 200 GB of data, 1,000 RPS demand, and cache nodes that each hold 10 GB and serve 100 RPS, ten *replicated* caches still serve the RPS but each holds the same 10 GB — 5% of the data. Ten *sharded* caches serve the same RPS but each holds a unique 10 GB — 50% coverage, a tenfold gain in working-set memory. But sharded caches are fragile: each user always maps to the same shard, so losing a shard means that user permanently misses cache until restoration. Capacity planning must account for this: a backend rated for 1,000 RPS fronted by a 50%-hit cache appears to do 2,000 RPS but should be sized at ~1,500 RPS to survive losing half the cache replicas. The remedy is to implement each shard as a small **replicated, sharded service** — combining replication with sharding so shards survive individual failures, peak-time rollouts become safe, and each shard can scale independently in response to its own load.

Choosing the **sharding function** — `Shard = ShardingFunction(Req)`, commonly `hash(Req) % N` — requires two properties: *determinism* (same input always yields the same shard) and *uniformity* (load spreads evenly). Picking the right *key* is harder. Hashing the whole request is wrong: request time and client IP would force two functionally identical requests to different shards. Hashing only the path is too general: French and English users asking for the same path land on the same shard. Hashing IP and path is too specific: two French IPs split across shards. The right level — `shard(country(request.ip), request.path)` — derives a coarse attribute from the IP so locale-equivalent requests share a shard. Picking the right key requires understanding which request attributes actually change the response.

A naive `hash(Req) % N` has a fatal property: changing N from 10 to 11 remaps a large fraction of keys, approximating a complete cache failure. A **consistent hashing function** guarantees only roughly *# keys / # shards* are remapped when the shard count changes — under 10% for a 10→11 scale-up. Burns demonstrates this with `nginx`'s `hash $request_uri consistent` directive.

Sharding extends beyond caches. A multi-player game whose virtual world exceeds one machine's capacity can shard by player location, since players far apart in the world rarely interact. **Hot sharding** addresses an inevitability: organic traffic skew (one viral photo) drives disproportionate load to a single shard. With replicated shards plus autoscaling, a hot shard gets more replicas while cold shards can be co-located on fewer machines, rebalancing dynamically as patterns shift.

### 3.3. Scatter/Gather

The **scatter/gather pattern** uses replication for scalability *in time*, farming a single request out to many leaves in parallel and merging their partial responses at the **root**. Where replicated patterns scale per-request and sharded patterns scale per-data, scatter/gather shards the work of *one* request — a fit for problems with large amounts of mostly-independent computation per request.

The simplest variant is **root distribution** with replicated leaves: every leaf holds the same data, and the root distributes pieces of work across leaves. Parallelizing across machines avoids the memory, network, and disk bandwidth bottlenecks that cap multi-threaded single-machine speedups, keeping CPU as the bottleneck. The book illustrates with distributed document search using an index built as a hashtable (word → list of documents): for "cat AND dog," the root farms one leaf per word and returns the *intersection* of the per-word document sets. **Leaf sharding** kicks in when total data exceeds one machine's memory or disk: data is partitioned across leaves, every request goes to *all* shards, each searches its slice, and the root takes the *union* of partial results.

Choosing the leaf count requires care. Parallelization gains are asymptotic — every leaf adds constant per-request overhead (parsing, HTTP, etc.) that eventually dominates the actual work. Worse, the root must wait for *all* leaves before responding, so overall latency is set by the slowest leaf — the **straggler problem**. Tail percentiles dominate: with a 99th-percentile latency of 2s on one leaf, scattering to 5 leaves makes 2s a 95th-percentile event (0.99⁵ ≈ 0.95); to 100 leaves, 2s becomes essentially guaranteed. Availability suffers symmetrically: a 1% per-leaf failure rate over 100 leaves means nearly every user request fails. Because every shard is needed for every request, single-replica shards are fragile — and adding shards alone cannot scale compute since more shards worsen stragglers. The fix is the same as for sharded caches: replicate each leaf shard so the root load-balances across that shard's healthy replicas, masking failures and enabling rolling upgrades under load.

## 4. Event-Driven and Ownership Patterns

Two patterns sit alongside the long-running serving patterns: functions-as-a-service for short-lived event handling, and ownership election for the few stateful roles that must be held by exactly one replica.

### 4.1. Functions and Event-Driven Processing

**Functions as a service (FaaS)** is a request- or event-driven application style where short-lived code instances are spun up to handle a single request or event. Burns is careful to distinguish *serverless* from *event-driven*: a multi-tenant container-as-a-service is serverless but not event-driven; an open-source FaaS running on physical machines that operators administer is event-driven but not serverless. FaaS is typically one piece of a larger architecture, not a complete replacement for long-running services.

FaaS shines when its constraints align with the workload. Its benefits — the shortest path from code to running service (no artifact to push beyond source), automatic scaling and recovery, and forced statelessness that drives modular design — pay off most for stateless transformations and asynchronous events. But forced decoupling cuts both ways: each function communicates only over the network and must externalize all state to a storage service, and pathologies like an unintended cycle (`functionA → functionB → functionC → functionA`) become hard to detect and only stop when the original request times out or the bill becomes unpayable. Adopters must invest in rigorous monitoring and alerting, which adds friction that offsets FaaS's deployment simplicity. FaaS is a poor fit for **background processing** with time-bounded runtimes (video transcoding, log compression), for **in-memory data needs** where functions spin up while the user waits and large indices cause perceived latency spikes, and for **sustained request volume** where per-request pricing scales linearly while VM costs decrease per-core and via reservations.

Two FaaS-native patterns recur. The **decorator pattern** wraps an HTTP service with stateless functions that transform requests before they reach the service or responses before they return — for example, a request-defaulting decorator that fills in a `null` color field before forwarding to the API. Packaging the decorator as a sidecar adapter would couple its scale to the API; as a FaaS, the lightweight decorator scales independently and can be experimented with cheaply before being absorbed into the service. **Event handling** suits FaaS naturally because events are independent, stateless, highly variable in rate, and frequently new types are added. A two-factor authentication function fired from a login event generates a code, registers it, and texts the user (via Twilio), keeping slow text-message latency off the login web server's critical path. **Event-based pipelines** chain such functions into a directed graph — for example, a code submission that triggers a build, then branches on success (file an approval ticket whose closure triggers production push) or failure (file a bug and terminate). Pipelines can splice in non-software steps (a human approving a Jira ticket) that don't fit a microservices model.

### 4.2. Ownership Election

Some tasks demand that *exactly one* process own them at any time — a particular shard's master, the active scheduler in a high-availability deployment, a singleton background job. A single-server design solves this trivially with in-process locks but caps both scalability (no replication) and reliability (failure means downtime). For higher availability, **distributed ownership** requires a protocol so exactly one replica is the master at any moment and a new master is elected if the current one fails. Burns calls this **master election** throughout the chapter.

Often it isn't needed. The simplest alternative is the singleton: a single replica that implicitly owns everything, with the orchestrator providing automatic restart on crash, restart on failed health check, and rescheduling onto another machine if its node fails. The arithmetic is sobering: a daily-crashing container that restarts in ~2 seconds yields roughly three to four nines, but whole-machine failure with a ~5-minute reschedule drops to two nines if every machine fails daily. Deployment is a dominant cost — a singleton cannot run old and new versions concurrently, so a 2-minute daily upgrade caps availability at two nines and an hourly upgrade drops below one nine. Singletons work for background asynchronous processing where simplicity beats high availability; master election is reserved for systems where four-plus nines really matter.

When master election is required, Burns advises building on a distributed key-value store (etcd, ZooKeeper, consul) rather than implementing Paxos or RAFT directly — the latter is an academic exercise, not production work. Two primitives suffice: **compare-and-swap (CAS)**, an atomic write conditional on the current value matching an expected value, and **time-to-live (TTL)** so a key auto-clears after expiry. From these one builds a distributed mutex (CAS `"1"` over `"0"` to acquire, `"0"` over `"1"` to release), blocks on it efficiently using the store's change-watch primitive instead of polling, and survives holder crashes by writing the lock with a TTL. A **watchdog timer** crashes the holding process if the TTL elapses before it explicitly unlocks, so work never outlives its lock.

TTL introduces a subtle bug. If Process-1's TTL expires, Process-2 takes the lock, then Process-1's late `unlock` fires and releases someone else's lock, and Process-3 grabs it. The **resource version** every store write returns fixes this: record it on lock and require both expected value and matching resource version on unlock so a stale holder can't release a successor's lock.

For roles that must persist for the lifetime of a process — not just a short critical section — the basic lock is extended with a `renew()` operation that CAS-updates the same value with a fresh TTL. The holder runs a background thread that renews every `ttl/2` seconds, reducing the risk that the lock accidentally expires due to timing subtleties, and the renewal loop defines a `handleLockLost()` action when renewal fails; in a container orchestrator the cleanest action is to terminate the process and let the orchestrator restart it as a secondary waiting for the lock to become free. Even with leases, an over-scheduled CPU stall can briefly leave a stale holder believing it owns the lock after another replica has taken over, so guarded code adds a local pre-check (`isLocked()` plus `lockTime + 0.75 * ttl`), the called service double-checks the recorded owner on every request, and the lock's resource version travels with each request so the receiver can reject stale ones whose successor has already executed.

## 5. Batch Computational Patterns

The third family handles batch computation — finite work over a finite dataset — using the same container-based building blocks. The chapters build from a single work queue to compositions of queues coordinated by events to whole MapReduce-style aggregations.

### 5.1. Work Queue Systems

A **work queue** is the simplest form of batch processing: a batch of fully independent items processed without interaction. The goal is to ensure each item is processed within a target time, scaling workers up or down to keep pace with arrivals. Burns frames the system as a small set of reusable library containers parameterized by two application-specific interfaces. The **source container** surfaces work items to be processed and is an instance of the ambassador pattern — the generic queue manager is the application container, and the source container ambassadors out to the real-world queue (cloud storage listing, network share, Kafka or Redis topic). It exposes a versioned HTTP REST API on localhost — `GET /api/v1/items` returns an `ItemList`, `GET /api/v1/items/<item-name>` returns a single `Item` whose `data` field carries the per-item payload — and intentionally doesn't track completion, leaving that bookkeeping to the manager.

The **worker container** is started in its own container group via the orchestrator and receives exactly one invocation per item. Its API is file-based: it reads work from `WORK_ITEM_FILE` (in Kubernetes implemented by mounting a `ConfigMap` carrying the item's `data`). File-based works because only one call per worker is needed; an HTTP endpoint would have to defend against malicious or accidental injection from other cluster users. The shared work queue manager repeatedly lists source items, lists outstanding `Job` objects for the queue, diffs them, and creates a Kubernetes `Job` per missing item. Leaning on the orchestrator for durability means the manager doesn't need its own database for in-flight work; `Job` annotations carry per-item state.

Dynamic scaling needs care. Processing every item the moment it arrives can spike resource use; capping parallelism flattens load at the cost of higher per-item latency. The fundamental constraint is that average processing time divided by parallelism must beat the **interarrival time** — the average gap between items — or the backlog grows without bound. Three regimes: arrival 1/min and processing 30s leaves slack for bursts; 1/min and 1 min is balanced but cannot absorb sustained spikes; 1/min and 2 min always loses ground. The autoscaling rule of thumb: increase parallelism so effective processing time stays below interarrival time, and scale down only until effective processing time reaches roughly 90% of interarrival time, preserving a safety margin.

The **multi-worker pattern** specializes the adapter pattern: a wrapper container exposes the single worker interface but delegates the actual work to a composed group of reusable worker containers. A face-detection-then-blurring pipeline can reuse the blurring container in a separate car-detection-then-blurring pipeline, gaining code reuse while keeping the single-call worker contract intact.

### 5.2. Event-Driven Batch Processing

A **workflow system** chains work queues into a directed acyclic graph: the output of one queue becomes the input to one or more downstream queues, coordinated by completion events. Burns names five patterns for how queues link.

- **Copier** — duplicates a stream into multiple identical streams (one source video copied into separate render queues for a 4-KB high-resolution hard-drive format, a 1080-pixel digital-streaming format, a low-resolution format for mobile users on slow networks, and an animated GIF thumbnail).
- **Filter** — drops items that don't meet a criterion (out of newly signed-up users, keep only those who ticked the email-promotions box). Implemented as an ambassador wrapping an existing source.
- **Splitter** — like a filter but routes items to different queues based on the criterion so nothing is lost (shipped orders split into "email" and "text" queues; a splitter that sends the same input to multiple queues also acts as a copier).
- **Sharder** — generalized splitter that distributes a queue evenly across N shards via a sharding function, providing reliability via blast-radius containment, resource distribution across regions/datacenters, and dynamic rebalancing when a shard becomes unhealthy.
- **Merger** — inverse of the copier: combines multiple queues into one, modeled as a multi-source adapter (many source repositories funnel into one shared build-and-test queue).

Connecting these patterns benefits from a **publisher/subscriber API**: named queues called *topics* to which publishers write messages and from which subscribers receive them, with the queue reliably storing and delivering. Local-directory queues are confined to one node, and network filesystems add code and deployment complexity, so a managed pub/sub layer is simpler and scales horizontally. Choices include Azure EventGrid, Amazon SQS, and the open-source Apache Kafka, which is portable across on-prem hardware and cloud VMs. Topics typically map to workflow stages — a Photos sharder with three shards uses topics `Photos-1`, `Photos-2`, `Photos-3`. When creating a Kafka topic, `--replication-factor` (recommended 3 or 5) controls redundancy across machines, and `--partitions` caps how many machines a topic can spread across for load balancing.

### 5.3. Coordinated Batch Processing

Some batch outputs need aggregation. Burns frames this as the *reduce* phase of the MapReduce model: sharding work is the map; coordinating outputs into a single answer is the reduce. Two patterns cover the space.

**Join (barrier synchronization)** runs work in parallel but releases nothing past the join until *all* upstream work has completed — analogous to joining a thread. The merger pattern from the previous chapter blends two streams but gives no completeness guarantee, so it can't support correctness-critical aggregates like "sum every value in the set." Join provides that guarantee at a cost: it serializes the workflow at the barrier, increasing overall latency.

**Reduce** optimistically merges parallel outputs into a single comprehensive representation. Each reduce step takes two or more outputs and produces one of the same shape, so it can be applied iteratively until a single result remains, and because partial outputs share a type, reduce can run *concurrently* with the still-executing map phase. The book illustrates with three operations: counting word occurrences in a book by sharding pages across 10 work queues by the page-number's last digit, with each worker emitting per-word counts and reduce summing matching keys; computing total US population by sharding by state, then re-sharding by county, with workers emitting `(town, population)` tuples and reduce repeatedly combining pairs; and merging weighted **histograms** by multiplying each town's per-bucket percentages by population to recover counts, summing across towns, and dividing by merged population to renormalize.

A capstone hands-on combines the patterns into an image tagging and processing pipeline: license-plate detection plus blurring as a multi-worker container group sharded for throughput, a join barrier so originals aren't deleted until every image is blurred, a copier fanning into a deletion queue and a vehicle/color recognition queue, then a reduce stage summing per-image JSON tuples into a final aggregate count.

## 6. Conclusion: A New Beginning

Burns closes by tying the patterns to a broader trend. Every company is becoming digital — APIs and services must be delivered to mobile apps, IoT devices, and autonomous systems, demanding redundancy, fault tolerance, and high availability alongside rapid iteration. The need for distributed systems has grown by an order of magnitude, yet building them remains too costly and the pool of skilled engineers too small. Past leaps in software — compilers, high-level languages, object-oriented programming, managed code — widened the developer pool by crystallizing expert knowledge into reusable patterns and abstractions. Containers and orchestration paired with the patterns in this book, Burns argues, are the next such leap. The patterns themselves — sidecars, ambassadors, sharded services, FaaS, work queues, and the rest — form a shared vocabulary, but the real payoff comes when developers collaborate on canonical reusable implementations rather than rebuilding from scratch.

## 7. Key Takeaways

- **Patterns democratize distributed systems.** Just as algorithmic patterns and OO design patterns widened those developer populations, container-based distributed-system patterns aim to do the same — turning a black art into a shared practice.
- **Containers are the abstraction that makes the patterns real.** Coscheduled container groups (e.g., Kubernetes pods) let two containers share namespaces tightly enough to compose at runtime without source-level integration.
- **Sidecar, ambassador, adapter** — three single-node roles for a second container: augment the application, broker its outbound traffic, or normalize its inbound interface.
- **Reusable means parameterized, API-defined, and documented.** Treat each container as a function with explicit inputs, a stable contract for callers, and Dockerfile-level docs (`EXPOSE`, `ENV`, `LABEL`).
- **Replicated, sharded, scatter/gather** — three scaling axes for serving: more requests, more state, or more compute per request. The patterns combine; replicated-sharded is the standard answer to fragility in pure sharded systems.
- **Edge layers add value beyond load balancing.** Caching, rate limiting, and SSL termination belong in the load-balanced tier; cache layers should be few and fat, and session affinity moves from IPs to cookies once caches sit in the path.
- **FaaS fits short, stateless, event-driven work.** Avoid it for background processing, in-memory data, or sustained request volume — and budget for monitoring, since forced decoupling makes interaction bugs hard to find.
- **Master election is hard; the book recommends avoiding it when possible.** A singleton plus orchestrator restart often suffices. When distributed ownership is required, build on a key-value store's compare-and-swap and TTL primitives, use leases with `ttl/2` renewals, and carry resource versions through both unlock and downstream requests.
- **Batch composition follows the same playbook as serving.** Reusable library containers parameterized by source and worker interfaces; copier/filter/splitter/sharder/merger to wire queues; join and reduce to aggregate; pub/sub (Kafka) to connect everything.
- **Demand far exceeds supply.** The book's premise is that patterns and shared components will enlarge the population of engineers capable of building reliable distributed systems, paralleling earlier transitions in software development.
