## 1. Problem and Scope

- **Hotel reservation system** — booking backend for a chain like Marriott; same techniques transfer to Airbnb, flight reservations, and movie ticket booking.
- **In-scope features** — hotel detail page, room detail page, reserve a room, admin CRUD, **10% overbooking** (chain sells more rooms than it has, betting on cancellations).
- **Out of scope** — search; pricing varies by day (dynamic pricing) but the design just exposes daily rates.
- **Scale** — 5,000 hotels, 1M total rooms; 70% occupancy with 3-day average stay → ~240,000 reservations/day → **~3 reservations/sec** average. Reservation TPS is low; the challenge is concurrency on hot rooms during peak events.
- **Funnel QPS** — assuming 10% step-through and no prefetch: detail page ~300 QPS, booking page ~30 QPS, reservation ~3 TPS (working backwards from the funnel).
- **Non-functional priorities** — high concurrency on contested rooms, moderate latency (a few seconds is acceptable for the reservation request itself).

## 2. APIs and Microservice Architecture

- **REST endpoints in three groups** — hotel CRUD (`/v1/hotels/...`), room CRUD (`/v1/hotels/{id}/rooms/...`), reservations (`GET/POST/DELETE /v1/reservations`). Hotel and room mutations are admin-only.
- **Reservation request body** — includes `startDate`, `endDate`, `hotelID`, `roomTypeID` (not roomID — see data model), `roomCount`, and a client-supplied **`reservationID`** that doubles as the **idempotency key** to prevent double booking.
- **Microservice components** — CDN for static assets, public **API Gateway** (rate limiting, auth, route by endpoint), separate internal API behind VPN for staff, and services: **Hotel Service** (static hotel/room info, easily cached), **Rate Service** (per-day room rates), **Reservation Service** (bookings + inventory), **Payment Service** (executes payment, sets reservation `paid`/`rejected`), **Hotel Management Service** (admin operations).
- **Reservation status state machine** — `pending → paid → refunded`, plus `canceled` and `rejected` terminal states.
- **Inter-service RPC** — production systems typically use a high-performance framework like **gRPC**.

## 3. Data Model

- **Relational database chosen** — read-heavy workflow (orders of magnitude more browsing than booking), needs **ACID** to prevent negative balance / double charge / double booking, and the entity relationships (hotel → room_type → room) are stable and easy to model.
- **Reserve a room *type*, not a specific room** — hotels assign room numbers at check-in, unlike Airbnb where `listing_id` is fixed at booking. The schema therefore centers on `room_type_inventory`, not individual rooms.
- **`room_type_inventory` table** — composite primary key `(hotel_id, room_type_id, date)`, one row per day; columns `total_inventory` (rooms minus those off-market for maintenance) and `total_reserved`. Pre-populated 2 years ahead by a daily scheduled job.
- **Storage estimate** — 5,000 hotels × 20 room types × 2 years × 365 days = **73M rows**; fits a single DB. Use cross-region/AZ replication for HA rather than sharding.
- **Inventory check pseudocode** — `SELECT date, total_inventory, total_reserved FROM room_type_inventory WHERE …` over the date range, then per-row check `(total_reserved + numberOfRoomsToReserve) <= 110% * total_inventory` to enforce the overbooking rule.
- **If reservation data outgrows one DB** — archive past reservations to cold storage; shard active data by `hash(hotel_id) % N` (every common query already filters by `hotel_id`).

## 4. Concurrency Control

- **Two distinct double-booking problems** — (1) the same user double-clicks "book"; (2) multiple users race for the last room.
- **User double-click — idempotency key** — client UI gray-out is unreliable (JS disabled, accidental double-tap, network retry). The server-side fix uses `reservation_id` as the **primary key** of the `reservation` table; the second insert violates the unique constraint and is rejected. The reservation_id is generated by a globally unique ID generator at the time the order is created (before submit), and is included in the submit request.
- **Multi-user race example** — at non-serializable isolation, two transactions both read `total_reserved=99`, both pass the `<= total_inventory` check, both write `100`, both commit; one too many rooms get sold. The general fix is some form of locking.

| Option | How it works | Pros | Cons | Verdict |
|---|---|---|---|---|
| **Pessimistic locking** | `SELECT … FOR UPDATE` locks the row; later transactions wait | Simple under heavy contention; serializes updates | Deadlocks possible; long-held locks tank throughput | Not recommended |
| **Optimistic locking** | Add a `version` column; increment on update; reject if version doesn't match | Fast under low contention; no DB lock | Many failed retries under high contention → bad UX | **Good fit** — reservation QPS is low |
| **Database constraints** | `CHECK (total_inventory - total_reserved >= 0)` rolls back violations | Easy; no app-level logic | Same retry problem under contention; constraints not version-controlled with code; not all DBs support them | **Good fit** — easy to implement at this scale |

- **Why optimistic locking fits hotels** — reservation QPS is naturally low (~3/sec), so version-check failures are rare and the absence of DB locks lets reads scale.

## 5. Scaling the System

- **Stateless services scale horizontally** — the bottleneck is the database, which holds all state.
- **Database sharding by `hotel_id`** — every key query filters by hotel; spreading 30,000 QPS across 16 shards yields ~1,875 QPS per shard, well within MySQL's capacity.
- **Booking.com / Expedia scenario** — 1,000× higher load than a single chain forces sharding plus a cache layer; without it the inventory DB becomes the bottleneck.

## 6. Cache Layer and Consistency

- **Inventory cache (Redis)** — key `hotelID_roomTypeID_{date}`, value = available room count; reads served by Redis since "check inventory" outnumbers "reserve" by an order of magnitude. Use **TTL** + **LRU eviction** to drop historical inventory automatically (only current/future inventory is meaningful).
- **Cache-DB sync** — DB is the source of truth; cache is updated **asynchronously**. Two propagation paths: application code writes the cache after DB commit, or **Change Data Capture (CDC)** via tools like **Debezium** reads DB changes and applies them to Redis.
- **Eventual consistency is acceptable** — the DB does the **final inventory validation** on every reserve. If the cache stales out and shows availability while the DB is full, the user gets a "someone just booked it" error and a refresh shows the correct state. The reverse staleness merely delays a booking the user could have made. Crucially, **never trust the cache alone** to commit a reservation.
- **Pros / cons** — pros: drastically reduced DB load, fast in-memory reads. Cons: cache-DB consistency is genuinely hard and must be reasoned about explicitly.

## 7. Cross-Service Data Consistency

- **Hybrid pragmatic choice** — Reservation Service owns **both** reservation and inventory tables in the same relational DB, so a single ACID transaction covers a booking. This deliberately bends "one DB per microservice" purism in exchange for simpler correctness.
- **Pure microservices problem** — each service owns its DB, so a logically atomic booking spans multiple DBs and can't use one transaction; partial failures leave inventory and reservation tables out of sync.
- **Industry options for cross-service atomicity**:
  - **Two-phase commit (2PC)** — atomic across nodes, but **blocking**: any participant failure stalls progress until recovery; not performant.
  - **Saga** — sequence of local transactions, each emitting a message to trigger the next; failures trigger **compensating transactions** to undo prior steps. Eventual consistency only.
- **Author's stance** — the added complexity isn't worth it for this problem; co-locating reservation and inventory under one relational DB is the more practical engineering choice.
