# Hotel Reservation System

> Vol 2 only — neither Liu nor Xu Vol 1 covers this. ## 1. Requirements

Booking backend for a chain like Marriott. Same techniques transfer to Airbnb, flights, movie tickets.

- In scope: hotel detail page, room detail page, reserve a room, admin CRUD, **10% overbooking** (chain sells more rooms than it has, betting on cancellations).
- Out of scope: search; pricing varies by day but the design just exposes daily rates.
- Scale: 5,000 hotels, 1M total rooms; 70% occupancy with 3-day average stay → ~240,000 reservations/day → **~3 reservations/sec** average. Reservation TPS is low; the challenge is **concurrency on hot rooms** during peak events.
- Funnel QPS — detail page ~300 QPS, booking page ~30 QPS, reservation ~3 TPS (10% step-through).

## 2. APIs and Microservices

REST endpoints in three groups:

```
/v1/hotels/...                        # admin-only CRUD
/v1/hotels/{id}/rooms/...             # admin-only CRUD
GET/POST/DELETE /v1/reservations
```

**Reservation request body** includes `startDate`, `endDate`, `hotelID`, `roomTypeID` (not `roomID` — see data model), `roomCount`, and a client-supplied **`reservationID`** that doubles as the **idempotency key**. Generated by a globally unique ID generator at the time the order is created, before submit. See [ID Generation](fundamentals/id-generation.md).

**Microservice components** — CDN for static assets, public **API Gateway** (rate limiting, auth, route by endpoint), separate internal API behind VPN for staff. Services: **Hotel Service** (static info, easily cached), **Rate Service** (per-day rates), **Reservation Service** (bookings + inventory), **Payment Service** (executes payment, sets reservation `paid`/`rejected`), **Hotel Management Service** (admin operations).

**Reservation status state machine** — `pending → paid → refunded`, plus `canceled` and `rejected` terminal states.

**Inter-service RPC** — production systems typically use a high-performance framework like **gRPC**.

## 3. Data Model

**Relational chosen** — read-heavy workflow (orders of magnitude more browsing than booking), needs **ACID** to prevent negative balance / double charge / double booking, and the entity relationships are stable and easy to model.

**Reserve a room *type*, not a specific room** — hotels assign room numbers at check-in (unlike Airbnb where `listing_id` is fixed at booking). The schema therefore centers on `room_type_inventory`, not individual rooms.

**`room_type_inventory` table** — composite primary key `(hotel_id, room_type_id, date)`, one row per day:

| Column | Notes |
|---|---|
| `total_inventory` | Rooms minus those off-market for maintenance |
| `total_reserved` | Currently reserved count |

Pre-populated 2 years ahead by a daily scheduled job.

**Storage estimate** — 5,000 hotels × 20 room types × 2 years × 365 days = **73M rows**; fits a single DB. Use cross-region/AZ replication for HA rather than sharding.

**Inventory check** — `SELECT date, total_inventory, total_reserved FROM room_type_inventory WHERE …` over the date range, then per-row check `(total_reserved + numberOfRoomsToReserve) <= 110% * total_inventory` to enforce the **overbooking** rule.

**If reservation data outgrows one DB** — archive past reservations to cold storage; shard active data by `hash(hotel_id) % N` (every common query already filters by `hotel_id`).

## 4. Concurrency Control

Two distinct double-booking problems:

1. **User double-clicks "book"** — client UI gray-out is unreliable (JS disabled, accidental double-tap, network retry). Server-side fix: use `reservation_id` as the **primary key** of the `reservation` table; the second insert violates the unique constraint and is rejected. See [Concurrency & Transactions §7](fundamentals/concurrency-and-transactions.md#7-idempotency-keys-as-primary-keys).
2. **Multi-user race** — at non-serializable isolation, two transactions both read `total_reserved=99`, both pass the `<= total_inventory` check, both write `100`, both commit; one too many rooms get sold. Needs locking.

| Option | Pros | Cons | Verdict |
|---|---|---|---|
| **Pessimistic locking** (`SELECT … FOR UPDATE`) | Serializes updates; simple under heavy contention | Deadlocks; long-held locks tank throughput | Not recommended |
| **Optimistic locking** (version column) | Fast under low contention; no DB lock | Many failed retries under high contention → bad UX | **Good fit** — reservation QPS is low |
| **DB constraints** (`CHECK (inventory - reserved >= 0)`) | Easy; no app-level logic | Same retry problem; constraints not version-controlled with code; not all DBs support them | **Good fit** — easy at this scale |

**Why optimistic fits hotels** — reservation QPS is naturally low (~3/sec), so version-check failures are rare and the absence of DB locks lets reads scale. See [Concurrency & Transactions §4](fundamentals/concurrency-and-transactions.md#4-pessimistic-vs-optimistic-locking).

## 5. Scaling

Stateless services scale horizontally; the bottleneck is the database.

**Database sharding by `hotel_id`** — every key query filters by hotel; spreading 30,000 QPS across 16 shards yields ~1,875 QPS per shard, well within MySQL's capacity. See [Sharding & Consistent Hashing](fundamentals/sharding-and-consistent-hashing.md).

**Booking.com / Expedia scenario** — 1,000× higher load than a single chain forces sharding plus a cache layer; without it the inventory DB becomes the bottleneck.

## 6. Cache Layer and Consistency

**Inventory cache (Redis)** — key `hotelID_roomTypeID_{date}`, value = available room count. Reads served by Redis since "check inventory" outnumbers "reserve" by an order of magnitude. **TTL** + **LRU eviction** drops historical inventory automatically.

**Cache-DB sync** — DB is the source of truth; cache is updated **asynchronously**. Two propagation paths: application code writes the cache after DB commit, or **Change Data Capture (CDC)** via tools like **Debezium** reads DB changes and applies them to Redis.

**Eventual consistency is acceptable** because the DB does the **final inventory validation** on every reserve. **Cache lies; the DB tells the truth.** If the cache stales out and shows availability while the DB is full, the user gets a "someone just booked it" error and a refresh shows the correct state. The reverse staleness merely delays a booking the user could have made. **Never trust the cache alone** to commit a reservation.

## 7. Cross-Service Data Consistency

**Hybrid pragmatic choice** — the Reservation Service owns **both** reservation and inventory tables in the same relational DB, so a single ACID transaction covers a booking. This deliberately bends "one DB per microservice" purism in exchange for simpler correctness.

**Pure microservices alternative** — each service owns its DB, so a logically atomic booking spans multiple DBs and can't use one transaction. Industry options:

- **Two-phase commit (2PC)** — atomic across nodes but **blocking**. See [Concurrency & Transactions §6.1](fundamentals/concurrency-and-transactions.md#61-two-phase-commit).
- **Saga** — sequence of local transactions with compensating rollbacks; eventual consistency only. See [§6.6](fundamentals/concurrency-and-transactions.md#66-saga).

Author's stance: the added complexity isn't worth it for this problem; co-locating reservation + inventory under one relational DB is the more practical engineering choice.
